# 引论：GIT 的正向叙事与逆向缺口

GIT 框架的核心公理 $L(X) = L(G) + L(R|G) \to \min$ 描述了一个从需求到实现的正向过程：系统通过发现共享生成规则 $G$ 来压缩残差 $R$，在扩张-收敛的呼吸节律中逐步逼近最小描述长度。

但软件工程中存在一类同等重要的活动被现有理论忽略：**逆向工程**——面对一个已存在但不可理解的系统制品（混淆后的代码、压缩后的 bundle、遗留系统、第三方闭源组件），恢复其结构、意图和生成规则。

Declaude 项目提供了一个精确的实例：将 Claude Code 的 12MB minified esbuild bundle 还原为 119 个语义命名的 TypeScript 模块、150+ 个 TypeScript 接口、9 层架构，最终达到 V_instance = 0.807、V_meta = 0.811 的质量评分。这个过程不是 GIT 正向理论的简单逆运用——它揭示了一套独立的信息论结构，值得作为 GIT 框架的**对偶理论**进行系统阐述。

# 一、两种压缩、两种最优：机器 MDL 与认知 MDL 的分离

## 1.1 esbuild 的字面 MDL 最小化

esbuild 对 JavaScript/TypeScript 代码的打包过程是 GIT 基本公理的字面实现：

- **变量名缩短**：`SdkBridgeApi` → `BRq`，直接降低 $L(R|G)$ 中标识符的编码长度
- **模块边界抹平**：将 119 个独立文件合并为单一 bundle，消除模块声明和导入语句的开销
- **死代码剪枝**：tree-shaking 移除未使用路径，降低 $L(R)$
- **Wrapper 注入**：`E()/x()` 模块封装替代原始的 ESM import/export

最终产物是一个对机器而言描述长度接近最优的制品——它在通用图灵机上执行的效率极高，但对人类而言几乎完全不可理解。

## 1.2 两个不同的生成规则 $G$

这揭示了 GIT 当前表述中一个隐含的假设：MDL 最小化中的 $G$（共享生成规则）对机器和人类是同一个 $G$。Declaude 的实践表明这个假设不成立。

**机器的 $G_M$**：JavaScript 运行时、V8 引擎的字节码格式、操作系统的系统调用接口。在 $G_M$ 下，`BRq` 和 `SdkBridgeApi` 的描述长度差异仅在于标识符的字符数——功能等价。

**人类的 $G_H$**：编程语言的命名惯例、模块化设计模式、类型系统的语义、软件架构的通用模式（MVC、事件驱动、中间件管道等）。在 $G_H$ 下，`SdkBridgeApi` 编码了"这是一个 SDK 桥接层的 API"这一语义信息，而 `BRq` 需要额外的上下文信息才能解码——即 $L(R|G_H)$ 远大于 $L(R|G_M)$。

形式化地，存在两个不同的 MDL 优化目标：

$$L_M(X) = L(G_M) + L(R|G_M) \to \min \quad \text{（机器 MDL）}$$

$$L_H(X) = L(G_H) + L(R|G_H) \to \min \quad \text{（认知 MDL）}$$

esbuild 优化的是 $L_M$；Declaude 优化的是 $L_H$。两者在同一个功能等价约束 $T(X) = \text{pass}$ 下操作，但其最优解位于程序空间 $\mathcal{P}$ 中的不同位置。

## 1.3 认知 MDL 的特殊结构

认知 MDL $L_H$ 与机器 MDL $L_M$ 的关键差异在于 $G_H$ 的结构。人类的认知先验 $G_H$ 包含若干 $G_M$ 中不存在的分量：

- **语义命名惯例**：变量名应反映其角色（如 `permissionMode` 而非 `pM`）
- **模块边界**：功能内聚的代码应聚集为命名模块，模块间通过显式接口通信
- **类型注解**：参数和返回值的类型约束应显式标注
- **层级结构**：系统应可分解为有限数量的有意义层级（如 CLI / Session / Agent Loop / Tools / API）
- **文档与注释**：关键设计决策应以自然语言记录

这些分量在 $G_M$ 的视角下是**纯冗余**——它们不改变程序的执行行为，只增加描述长度。但在 $G_H$ 的视角下，它们是**必要的解码上下文**——没有它们，人类无法在合理时间内理解系统的结构和意图。

这导出一个重要的理论修正：

> **GIT 双目标修正**：系统的总描述长度应同时考虑机器执行效率和人类认知效率。最优系统设计不是单纯的 $L(X) \to \min$，而是在 $L_M$ 和 $L_H$ 之间的帕累托前沿上选点。

这一修正与 GIT 原始框架中"实现谱系"（prompt → workflow → DSL → 代码）的讨论一致，但将其从实现形式的选择推广到了**同一实现形式内部**的表示优化。

# 二、收敛的方向反转：逆向工程的谱系

## 2.1 正向收敛与逆向收敛

GIT 的正向收敛谱系是：

$$\text{自然语言 prompt} \xrightarrow{\text{压缩}} \text{结构化 workflow} \xrightarrow{\text{压缩}} \text{DSL} \xrightarrow{\text{压缩}} \text{编译型代码}$$

Declaude 的逆向过程沿一条不同的谱系运行：

$$\text{压缩 bundle} \xrightarrow{\text{解压}} \text{模块化代码} \xrightarrow{\text{类型恢复}} \text{类型化代码} \xrightarrow{\text{语义恢复}} \text{可理解的系统}$$

两条谱系的共同约束是**行为不变性**：正向收敛要求测试 $T$ 持续通过；逆向收敛要求功能对等（golden file 测试、acceptance test 通过）。

但它们的优化方向完全相反：

| 维度 | 正向收敛 | 逆向收敛 |
| :---- | :---- | :---- |
| **$L(R\|G)$ 方向** | 递减 | 递增（增加认知冗余） |
| **$G$ 的变化** | 发现新 $G$，降低 $R$ | 恢复旧 $G$，使 $R$ 可理解 |
| **稳定性 $\mathcal{L}_S$** | 递减（行为更确定） | 不变（行为对等约束） |
| **本征维度 $d$** | 递减（抽象压缩） | 先升后降（先展开后重组） |

## 2.2 逆向收敛的四阶段

Declaude 的实践揭示了逆向收敛的典型阶段结构：

| 阶段 | 名称 | 操作 | 信息论特征 |
| :---- | :---- | :---- | :---- |
| **0** | 基线建立 | 建立 golden file、捕获原始行为、设定功能对等约束 | $L(A)$ 固化，$T(X) = \text{pass}$ 作为锚定 |
| **1** | 结构恢复 | 去除 wrapper、分离模块、恢复导入关系 | $L(G_H)$ 的"模块边界"分量开始生长 |
| **2** | 语义恢复 | 标识符重命名、类型注解、接口提取 | $L(G_H)$ 的"命名/类型"分量加速增长，$L(R\|G_H)$ 开始下降 |
| **3** | 架构理解 | 识别层级、发现设计模式、文档化关键决策 | 系统可用 ArchGuard 等工具度量，架构质量可量化 |

这个四阶段模型与 GIT 测试理论中的自举四阶段（§五）形成结构对偶：

| 测试自举 | 逆向自举 |
| :---- | :---- |
| 初始状态 ⊥（无测试） | 初始状态（不可读的 bundle） |
| LLM 临时锚定（虚拟测试层） | Golden file 锚定（行为基线） |
| 跨层下推（固化为断言） | 结构恢复（固化为模块） |
| 代谢完成（自持测试套件） | 架构理解（自持知识体系） |

## 2.3 逆向工程中的扩张-收敛呼吸

逆向过程本身也遵循 GIT 的呼吸节律，但其含义发生了偏移：

**逆向扩张**：探索未知结构。在 Declaude 中表现为：尝试不同的模块分割方案、追踪调用链以发现隐含的架构层级、引入外部知识源（GitHub issues、API 文档、类型定义文件）来推断语义。本征维度 $d$ 暂时上升。

**逆向收敛**：固化已发现的结构。在 Declaude 中表现为：确认模块命名、固定接口定义、通过构建验证确认模块划分的正确性。$d$ 回落。

Declaude 的 10 次迭代轨迹清晰展示了这种呼吸：

- Iteration 0-2：大规模扩张（依赖剥离、模块初始切分）
- Iteration 3-4：收敛（TypeScript 转换、构建验证）
- Iteration 5-6：再扩张（语义分析、外部知识引入）
- Iteration 7-9：深度收敛（专用 meta-agent 执行精细重命名和类型恢复）
- Iteration 10：最终收敛（V_instance ≥ 0.80, V_meta ≥ 0.80, ΔV < 0.02）

# 三、生成规则恢复：逆向 MDL 问题

## 3.1 形式化定义

GIT 的正向问题是：

$$\text{给定需求 } T, \quad \text{求 } G^*, R^* = \arg\min_{G,R} [L(G) + L(R|G)] \quad \text{s.t. } T(X(G,R)) = \text{pass}$$

逆向工程提出了一个对偶问题：

$$\text{给定压缩制品 } C = \text{compile}(G_{\text{original}}, R_{\text{original}}), \quad \text{求 } \hat{G}, \hat{R} = \arg\min_{G,R} [L_H(G) + L_H(R|G)] \quad \text{s.t. } \text{behavior}(X(G,R)) \equiv \text{behavior}(C)$$

注意两个关键差异：

1. 约束不是"通过测试 $T$"，而是"行为等价于 $C$"——后者是更强的约束，因为 $C$ 隐含了所有可能的行为，包括未被任何 $T$ 覆盖的边界情况。
2. 优化的是 $L_H$（认知 MDL）而非 $L_M$（机器 MDL）。

## 3.2 信道解码类比

逆向 MDL 问题在信息论中有一个精确的类比：**信道解码**。

将 esbuild 的打包过程视为一个有损信道：

$$G_{\text{original}}, R_{\text{original}} \xrightarrow{\text{esbuild（编码器）}} C \xrightarrow{\text{逆向工程（解码器）}} \hat{G}, \hat{R}$$

信道的"损失"体现在：

- **标识符信息丢失**：语义命名 → 单字母变量（不可逆的信息损失）
- **模块边界信息丢失**：显式 import/export → 内联引用（部分可恢复）
- **类型信息丢失**：TypeScript 类型注解 → 运行时 JavaScript（部分可从 `.d.ts` 恢复）
- **注释和文档丢失**：完全不可逆

Shannon 信道编码定理告诉我们：**恢复精度的上界取决于信道容量和冗余编码**。Declaude 的实践验证了这一点——标识符可读性仅恢复到 22.3%（216/970），因为标识符的语义信息在 minification 中被不可逆地擦除。但结构清晰度达到 100%，因为模块边界信息可以通过分析 wrapper 模式和调用图来恢复。

## 3.3 辅助信道与先验知识

信道解码的关键技术是利用**辅助信道**（side channels）来补偿主信道的信息损失。Declaude 使用的辅助信道包括：

| 辅助信道 | 恢复的信息类型 | 贡献 |
| :---- | :---- | :---- |
| `sdk-tools.d.ts`（2846 行） | 接口定义、参数类型、返回类型 | 类型覆盖从 0% → 68% |
| `@anthropic-ai/sdk` npm 包 | API 方法名、请求/响应结构 | 关键标识符的语义恢复 |
| GitHub issues & 文档 | 功能命名、设计意图 | 高层架构的语义标注 |
| LLM 先验知识 | 常见代码模式的统计先验 | 语义推断（如 `BRq` → Bridge/Request 相关） |

在 GIT 框架中，这些辅助信道对应于**额外的生成规则 $G_{\text{aux}}$**。逆向工程的质量直接取决于 $G_{\text{aux}}$ 的丰富度：

$$L_H(\hat{R}|\hat{G}) = L_H(\hat{R}|\hat{G}, G_{\text{aux}}) + \text{gap}(G_{\text{aux}})$$

其中 $\text{gap}(G_{\text{aux}})$ 是辅助信道未能覆盖的信息损失——这就是 Declaude 中标识符可读性停滞在 22.3% 的信息论根因。

## 3.4 恢复精度的不可计算性与实践近似

与正向 MDL 的 $K(x)$ 不可计算类似，逆向 MDL 的最优解也不可计算——因为我们无法精确知道 $G_{\text{original}}$ 是什么。但 Declaude 的实践表明，通过以下策略可以达到足够好的近似：

1. **结构优先于语义**：先恢复可机械判定的结构（模块边界、调用图），再恢复需要推断的语义（命名、文档）
2. **高频模式优先**：在 bundle 中出现频率最高的模式最有可能对应真实的架构结构
3. **外部锚定优先**：有 `.d.ts` 或 API 文档支撑的恢复优先于纯推断

这三条策略恰好对应 GIT 的 Fisher 信息采样原则：**优先在信息量最高（最确定）的方向上进行采样**。

# 四、表征测试的信息论地位

## 4.1 表征测试与规范测试的区分

GIT 的测试理论（见 [测试能力建设方法论](./git_framework_testing_capability_building_methodology.docx.md)）将测试建模为定义可行流形 $\mathcal{M}_T$ 边界的约束。但该理论隐含假设测试是**规范性的**——即测试表达的是"系统应当如何行为"。

Declaude 大量使用的**表征测试（Characterization Testing）**具有完全不同的信息论性质：它不表达"应当如何"，而是锁定"目前如何"。Michael Feathers 在《Working Effectively with Legacy Code》中系统性地提出了这一概念。

形式化地：

- **规范测试**：$T_{\text{spec}}(X) = \mathbb{1}[\text{behavior}(X) \in \text{Spec}]$ ——行为是否在规范范围内
- **表征测试**：$T_{\text{char}}(X) = \mathbb{1}[\text{behavior}(X) = \text{behavior}(X_0)]$ ——行为是否与基线 $X_0$ 一致

两者在 Fisher 信息几何中的位置截然不同。

## 4.2 Fisher 信息几何的差异

**规范测试**在程序空间中定义了一个**有界区域**（规范的容许范围）。Fisher 信息矩阵在该区域边界上取较高值——行为接近规范边界时，测试的判别力最强。这就是传统边界值分析的几何解释。

**表征测试**在程序空间中定义了一个**单点锚定**——当前行为。Fisher 信息矩阵在该点的所有方向上取较高值，因为任何偏离都会被检测。但这也意味着**表征测试没有方向偏好**——它不区分"好的偏离"和"坏的偏离"。

在逆向工程中，这种无偏性恰恰是优势。逆向工程者面对未知流形，不知道哪些行为变化是安全的、哪些是危险的。表征测试提供了一种**无需理解即可锁定**的机制：

$$T_{\text{char}}: \text{"我不知道为什么系统这样工作，但我确保它继续这样工作"}$$

## 4.3 从表征测试到规范测试的演化

Declaude 的实践揭示了一个渐进的测试演化路径：

1. **Golden file 阶段**（纯表征）：捕获 bundle 的完整输出，任何字节级差异都触发失败
2. **行为对等阶段**（松弛的表征）：允许输出格式的微小差异，但锁定功能语义
3. **契约测试阶段**（表征→规范过渡）：基于恢复的接口定义，为关键 API 编写接口契约测试
4. **架构测试阶段**（规范）：基于理解后的架构，编写模块边界和依赖方向的约束测试

这个演化路径对应于逆向工程者对系统理解的深化——从"不理解但锁定"到"理解并规范"。在 GIT 框架中，这对应于 Fisher 信息矩阵从**各向同性**（表征测试的无偏采样）到**各向异性**（规范测试的定向采样）的演化。

## 4.4 对 GIT 测试理论的扩展

将表征测试纳入 GIT 测试理论，需要扩展测试算子的分类学：

| 测试类型 | 探测 π | 解码 δ | 先验要求 | 适用阶段 |
| :---- | :---- | :---- | :---- | :---- |
| **规范测试** | 执行+断言 | 二值（规范内/外） | 需要明确的规范 | 正向开发 |
| **表征测试** | 执行+快照比较 | 二值（一致/偏离） | 无需理解系统 | 逆向工程、遗留系统 |
| **变异测试** | 注入变异+执行 | 杀伤/存活 | 需要已有测试套件 | 测试盲区检测 |
| **属性测试** | 随机输入+不变量检查 | 二值（不变量成立/违反） | 需要识别不变量 | 探索性验证 |

表征测试填补了 GIT 测试理论的一个空白：**在不具备系统先验知识的条件下锁定系统行为的能力**。这在遗留系统维护、第三方依赖升级、以及逆向工程中具有核心价值。

# 五、生产性自指：ArchGuard-Claude Code-Declaude 的三角环路

## 5.1 自指结构的实例化

GIT 完整框架（§13）将生产性自指定义为"元层审视自身，发现跨层共享结构，将其下推到对象层"。Declaude 项目实例化了一个三角自指环路：

```
ArchGuard（架构分析工具）
    ↓ 分析
Claude Code（被分析对象——AI 编码助手）
    ↓ 驱动
Declaude（逆向工程实验）
    ↓ 使用 ArchGuard 度量质量
ArchGuard（闭环）
```

具体而言：

1. **ArchGuard 分析 Declaude 的输出**：ArchGuard 对还原后的 Claude Code 源码执行架构分析（entity/relation/cycle 检测），产出架构质量得分（73.4%）。这是 ArchGuard 作为**对象层工具**的标准使用。

2. **Declaude 的输出丰富 ArchGuard 的元知识**：通过逆向工程发现的 Claude Code 架构（9 层、26 工具、6 权限模式、3 MCP 传输方式），成为 ArchGuard 未来分析 Agent 系统时可复用的**领域先验**。这是知识从对象层向元层的**上推**。

3. **GIT 理论指导 Declaude 的方法论**：Declaude 的价值函数设计（V_instance/V_meta）、迭代收敛策略、表征测试优先的验证方法，都可以用 GIT 理论来解释和优化。这是元层对对象层的**下推**。

## 5.2 跨层共享结构的发现

GIT 的创造性自指理论（§13.3）预测：自指结构能发现跨层共享结构，从而降低 $\hat{\mathcal{P}}$ 中的总描述长度。Declaude 验证了这一预测。

**发现的跨层共享结构示例**：

- **权限模型模式**：Claude Code 的 6 种权限模式（default, acceptEdits, bypassPermissions, dontAsk, plan, auto）是 Agent 系统的通用设计模式。ArchGuard 在未来分析任何 Agent 系统时，可以将这些模式作为 $G$ 的分量来检测和分类。

- **Agent Loop 架构**：`用户消息 → API 调用 → 工具执行 → 结果反馈 → 循环` 的模式不限于 Claude Code——它是所有 ReAct 架构 Agent 的共享结构。发现它降低了描述任何类似系统的 $L(R|G)$。

- **MCP 集成模式**：Model Context Protocol 的 stdio/SSE/HTTP 三传输方式是 Agent-Tool 耦合的通用接口标准。理解其在 Claude Code 中的实现方式，直接可复用于分析其他 MCP 兼容系统。

形式化地，Declaude 项目实现了：

$$L(\text{ArchGuard}') + L(\text{Agent 分析知识}') < L(\text{ArchGuard}) + L(\text{Agent 分析知识})$$

——跨层总描述长度下降，这正是生产性自指的定义。

## 5.3 盲点定理在自指环路中的体现

GIT 盲点定理（§12.2）指出：任何足够复杂的自指元层 $\mathcal{M}$，都存在它无法检测到的自身缺陷。Declaude 的三角环路中，盲区表现为：

- **ArchGuard 无法检测的 Claude Code 结构**：Claude Code 的某些行为依赖运行时动态分派（如 MCP server 的延迟加载），这些在静态分析中不可见
- **Declaude 无法恢复的原始意图**：标识符可读性 22.3% 的上限反映了 minification 信道的不可逆信息损失
- **GIT 理论无法自我验证的部分**：GIT 用自身的术语描述自身的局限——这本身就是一种自指，受到同样的盲点约束

这些盲区不是缺陷，而是自指结构的必然属性。GIT 框架的价值在于**明确标注盲区的位置**，使工程实践能够通过**外部锚定**（人类审计、交叉验证、互补工具）来补偿。

# 六、价值函数作为元反馈算子的工程化

## 6.1 反馈算子的四元组实例化

GIT 反馈理论将反馈机制形式化为四元组 $(π, δ, S, τ)$。Declaude 的两个价值函数——V_instance（实例质量）和 V_meta（方法论质量）——是这个理论的精确工程实例：

**V_instance 反馈算子**：

- **探测 π**：静态分析（AST 解析、ArchGuard 架构检测）+ 构建验证（esbuild 编译成功）+ 测试执行（golden file 对比、acceptance test）
- **解码 δ**：六维加权评分——可读性 (15%)、类型覆盖 (15%)、可构建性 (20%)、功能对等 (20%)、测试基础设施 (15%)、架构质量 (15%)
- **作用空间 S**：单次逆向迭代的输出制品
- **时间尺度 τ**：每次迭代（小时级）

**V_meta 反馈算子**：

- **探测 π**：方法论文档审计、迭代日志回溯、跨迭代对比
- **解码 δ**：五维加权评分——完整性 (20%)、有效性 (20%)、可复用性 (25%)、验证 (15%)、Agent 演化 (20%)
- **作用空间 S**：逆向方法论整体
- **时间尺度 τ**：每轮迭代审计（天级）

## 6.2 反馈层级对应

在 GIT 反馈理论的成熟度模型中，Declaude 的反馈结构对应于 **L2（多渠道协同）到 L3（元反馈驱动）的过渡阶段**：

- **L2 特征已达成**：V_instance 和 V_meta 形成两个互补的反馈通道。V_instance 提供"做得好不好"的信号，V_meta 提供"方法论在不在改进"的信号。两者的 Fisher 信息矩阵在不同维度上提供信息——V_instance 在代码质量维度上、V_meta 在方法论维度上。

- **L3 特征部分达成**：V_meta 本身就是二阶元反馈——它评估的不是逆向输出，而是评估方法本身的质量。但三阶元反馈（"评估 V_meta 的评估标准是否合理"）仅通过人类审计部分实现。

## 6.3 收敛判据与 Kleene 不动点

Declaude 的收敛条件——$V_{\text{instance}} \geq 0.80 \wedge V_{\text{meta}} \geq 0.80 \wedge \Delta V < 0.02$——直接对应 GIT 不动点理论中的 Kleene 不动点逼近：

- $V \geq 0.80$：不动点的吸引域进入阈值（"足够好"）
- $\Delta V < 0.02$：值收敛的判定条件（"不再显著改善"）

GIT 框架提醒我们：值收敛不等于点收敛。$\mathcal{M}^{(n)}$ 可能在等值面上做无限徘徊——Declaude 的渐近型收敛（Iteration 8-10 的微小振荡）正是这种模式的实例。

# 七、BAIME 方法论的 GIT 解释

## 7.1 方法论概述

Declaude 提出了 **BAIME（Bootstrap and Iterative Methodology Evolution）**作为逆向工程方法论。BAIME 的五个核心原则在 GIT 框架中有精确的理论对应：

| BAIME 原则 | GIT 理论对应 |
| :---- | :---- |
| 表征测试优先 | 在未知流形上的无偏 Fisher 采样（§4.2） |
| 增量转换+验证 | 行为不变约束下的逐步压缩（收敛阶段） |
| 架构优先 | 先恢复高层 $G$，再处理低层 $R$ |
| 知识驱动决策 | 利用辅助信道 $G_{\text{aux}}$ 补偿信息损失（§3.3） |
| 价值函数驱动 | 元反馈算子的显式编排（§6） |

## 7.2 "依赖剥离"失败的信息论解释

Declaude 最初采用的策略是"依赖剥离"（Dependency Peeling）：先外化第三方库，再分割第一方模块。这个策略失败了。

GIT 框架提供了精确的解释：依赖剥离是自底向上的——它试图先恢复最细粒度的 $R$，再推导 $G$。但逆向 MDL 的最优策略是自顶向下的——先恢复高阶结构 $G$（模块边界、架构层级），因为 $G$ 的恢复降低了所有后续 $R$ 恢复的条件描述长度：

$$L(R_i | G_{\text{recovered}}) \ll L(R_i | G_{\text{unknown}})$$

这与 GIT 正向理论中"从最低 $L(R|G)$ 的实现形式开始"的原则是对偶的：在逆向中，应当"从恢复最高信息增益的 $G$ 分量开始"。

## 7.3 Agent 演化的 GIT 解释

Declaude 观察到 Iteration 0-6 使用通用 Agent 处理，而 Iteration 7-10 需要专用 meta-agent（structural-transform、semantic-rename、type-recovery）。这对应 GIT 复杂度层级定律（§9）的直接预测：

> 单一元层试图同时负责审视、改进和评估时，其本征维度过高，会自发分裂为专用模块。

通用 Agent 在逆向前期够用——此时任务是粗粒度的结构切分，本征维度较低。但进入语义恢复阶段后，任务的本征维度升高（需要同时理解代码模式、命名惯例和领域语义），通用 Agent 的推理质量下降，迫使系统分化出专用 Agent。

# 八、对 GIT 理论的贡献与扩展

## 8.1 双目标 MDL

GIT 的基本公理应从单目标扩展为双目标：

$$\min \left[ L_M(X), L_H(X) \right] \quad \text{s.t. } T(X) = \text{pass}$$

在不同场景下，两个目标的权重不同：
- 生产部署：$L_M$ 主导（最小化执行和部署成本）
- 开发维护：$L_H$ 主导（最小化认知负荷）
- 逆向工程：纯 $L_H$ 优化（不改变 $L_M$）

## 8.2 逆向 MDL 问题的形式化

将以下问题纳入 GIT 理论框架：

$$\text{给定 } C = f(G_{\text{orig}}, R_{\text{orig}}), \quad \text{求 } \hat{G}, \hat{R} = \arg\min [L_H(\hat{G}) + L_H(\hat{R}|\hat{G})] \quad \text{s.t. } \text{behavior}(X(\hat{G},\hat{R})) \equiv \text{behavior}(C)$$

其中 $f$ 是编译/打包/混淆变换。这个问题的可解性取决于 $f$ 的信息保持特性——$f$ 越接近可逆，恢复精度越高。

## 8.3 表征测试的理论整合

将表征测试作为 GIT 测试分类学的第四类（与规范测试、变异测试、属性测试并列），特别关注其在以下场景中的信息论优势：

- 遗留系统的行为锁定
- 第三方依赖升级的回归保护
- 逆向工程的行为基线建立
- AI 生成代码的行为冻结

## 8.4 辅助信道理论

将逆向工程中利用外部知识源恢复信息的实践，形式化为 GIT 的辅助信道理论：

$$L_H(\hat{R}|\hat{G}) = L_H(\hat{R}|\hat{G}, G_{\text{aux}}) + I_{\text{lost}}(f) - I_{\text{recovered}}(G_{\text{aux}})$$

其中 $I_{\text{lost}}(f)$ 是变换 $f$ 造成的不可逆信息损失，$I_{\text{recovered}}(G_{\text{aux}})$ 是辅助信道恢复的信息量。

# 九、实践指导：基于 GIT 的逆向工程方法论

## 9.1 逆向工程成熟度模型

| 级别 | 名称 | 特征 | GIT 指标 |
| :---- | :---- | :---- | :---- |
| **L0** | 黑盒观察 | 仅通过输入-输出行为理解系统，无内部结构知识 | $L_H(X) = L_H(C)$（与原制品等同），Fisher 采样仅在输入-输出空间 |
| **L1** | 结构提取 | 模块边界已恢复，调用图已建立，但语义未知 | $G_H$ 的结构分量已恢复，$R$ 的标识符仍不可读 |
| **L2** | 语义恢复 | 关键标识符已重命名，类型注解已添加，架构层级已识别 | $L_H(R|G_H)$ 开始下降，ArchGuard 可产出有意义的度量 |
| **L3** | 完全理解 | 设计意图可解释，可独立修改和扩展，知识可迁移到其他系统 | $L_H(X) \approx L_H(X_{\text{original}})$，跨层知识下推已完成 |

## 9.2 Fisher 信息引导的逆向策略

1. **高 Fisher 区域优先**：首先攻克系统中行为变化最敏感的部分（入口点、核心循环、权限控制）——这些区域的恢复对总体理解的贡献最大
2. **辅助信道最大化**：穷尽所有可用的外部知识源（类型定义、API 文档、开源依赖的源码），最大化 $I_{\text{recovered}}$
3. **表征测试前置**：在任何结构变更前建立 golden file，锁定当前行为
4. **架构优先于细节**：先恢复高层 $G$（模块/层级/接口），再处理低层 $R$（实现细节/变量名）

## 9.3 反馈环路设计

借鉴 GIT 反馈理论的反馈编排原则：

1. **构建验证**（$τ$ = 秒级）：每次模块提取后立即验证 esbuild 编译通过
2. **Golden file 回归**（$τ$ = 分钟级）：每次结构变更后运行完整的行为对等测试
3. **ArchGuard 架构度量**（$τ$ = 小时级）：每个迭代结束后运行架构分析，跟踪 entity/relation/cycle 指标
4. **价值函数评审**（$τ$ = 天级）：每轮迭代结束后计算 V_instance/V_meta，评估收敛趋势

# 十、结论：压缩与解压是同一枚硬币的两面

## 10.1 核心论断

本文通过 Declaude 项目的实践分析，为 GIT 框架贡献了以下理论扩展：

1. **双目标 MDL**：系统的描述长度应区分机器 MDL ($L_M$) 和认知 MDL ($L_H$)。两者在同一行为等价约束下操作，但最优解位于程序空间的不同位置。逆向工程优化的是 $L_H$，与正向开发的 $L_M$ 优化形成对偶。

2. **逆向 MDL 的信道解码结构**：逆向工程在信息论上等价于信道解码——恢复精度受信道容量和辅助信道约束。标识符语义是不可逆损失；模块结构是可恢复信息；类型注解通过辅助信道（`.d.ts`）部分恢复。

3. **表征测试的独立地位**：表征测试不是规范测试的退化版本，而是在未知流形上进行无偏 Fisher 采样的独立工具。它在逆向工程和遗留系统维护中具有不可替代的价值。

4. **生产性自指的三角实例**：ArchGuard-Claude Code-Declaude 的三角环路验证了 GIT 的创造性自指理论——逆向工程过程中发现的架构知识可以反哺分析工具本身，降低跨层总描述长度。

## 10.2 对偶视角的统一意义

GIT 的正向理论回答"如何构建好的系统"；本文的对偶理论回答"如何理解已有系统"。两者共享同一套数学结构——MDL 最小化、Fisher 信息几何、扩张-收敛动力学、元层自指——但在不同的方向上应用。

这个对偶性的存在表明 GIT 不仅是一套工程方法论，而是一套关于**信息系统认识论**的更一般理论：

> 构建是压缩——从高熵的需求空间压缩为低熵的实现。理解是解压——从低可读性的制品解压为可理解的结构。两者都受到描述长度最小化原则的引导，都在程序空间的几何结构中寻找最优路径。

## 10.3 实践总结

| GIT 正向（构建） | GIT 对偶（理解） |
| :---- | :---- |
| 压缩：$L(X) \to \min$ | 反压缩：$L_H(X) \to \min$ |
| 生成规则发现 | 生成规则恢复 |
| 规范测试定义流形边界 | 表征测试发现流形位置 |
| 扩张→收敛（从 prompt 到代码） | 解压→重构（从 bundle 到模块） |
| 价值函数驱动迭代 | 价值函数驱动收敛判定 |
| 元层审视对象层 | 逆向工程产出反哺元层 |

**压缩即理解。解压亦理解。两条路径交汇于同一个信息几何。**
