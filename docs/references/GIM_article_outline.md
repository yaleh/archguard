# 几何信息方法（GIM）：统一人与 LLM 的软件开发操作模型

**Geometric Information Method: A Unified Operational Model for Human and LLM Software Development**

状态：提纲 v1
日期：2026-03-31
目标受众：使用 LLM 工具的软件架构师、LLM 辅助开发工具的构建者

---

## Thesis Statement

> 软件开发的所有活动——无论由人还是 LLM 执行——都是对同一约束空间的搜索与压缩操作。统一这个视角，可以给出 LLM 辅助开发中"何时扩张、何时收缩、何时固化、何时回退"的可操作判据。

---

## 0. 开场：一个具体的观察

*不做理论辩护。从一个可验证的事实开始。*

- 场景：将 ArchGuard 生成的架构图（ArchJSON → Mermaid）喂给 Claude Code，它在重构任务中做出了显著更好的决策——触碰更少的文件，保持更多的模块边界完整性
- 问题：为什么？架构图不包含任何代码——它是信息的**损失性压缩**。一个丢掉了大量细节的表示，为什么反而提升了决策质量？
- 回答的方向：因为软件开发不是"写代码"，而是在一个巨大的可能性空间中**搜索**。压缩后的结构表示缩小了搜索范围，过滤掉了不相关的维度。这对人和 LLM 同样成立
- **[案例 A]**：ArchGuard 自身重构的 before/after 数据——无架构上下文时 Claude Code 的变更扇出 vs. 有架构上下文时的变更扇出（从 git 历史提取具体数字）

## I. 核心论点：搜索、压缩与约束空间

*三个概念，用日常语言引入，不造新术语。*

### 1.1 约束空间

- 所有能通过测试、满足类型检查、符合业务规则的实现，构成一个"约束空间"（可行区域）
- 开发 = 在这个空间中找到一个点；重构 = 在功能等价的前提下移动到一个更好的点
- "更好"的含义：向新加入者（人或 LLM）解释系统所需的概念数更少
- 关键不是"谁在搜索"，而是搜索的特征——人类搜索带宽低但方向感强；LLM 搜索带宽高但容易漂移出约束空间（"幻觉"）

### 1.2 压缩即理解

- 好的抽象 = 捕获真实的共享结构，使总描述量减少
- 过度抽象 = 抽象本身的复杂度超过它消除的重复，总描述量反增
- 这不是新洞察（奥卡姆剃刀、YAGNI），但它提供了一个**统一的判据**：任何工程决策（抽象？重构？加测试？）都可以用"描述量是增还是减"来评估
- **描述量的代理观测**（不是精确数值，是可比较的序关系）：
  - 向新人/新 LLM 解释系统需要多少概念
  - 添加一个功能需要触碰多少文件
  - 存在多少"历史原因"的间接层
  - 移除一个组件后有多少其他组件需要修改

### 1.3 多目标性

- 同一系统存在多个正当的压缩目标：可理解性（语义命名、层级结构）vs. 执行效率（运行时性能、最小体积）
- 它们的最优解通常不重合：esbuild 的 minification 优化执行效率；架构工具优化可理解性
- 对 LLM 辅助开发的意义：LLM 消费的是**可理解性压缩**的产物，不是执行效率压缩的产物。架构图有用，minified 代码没用

## II. 核心贡献：收敛谱系

*这是 GIM 唯一真正原创的概念。给它最多篇幅。*

### 2.1 谱系定义

- 系统中的每个组件都处于一条连续谱系上——

  `统计推理（LLM/prompt） ⇄ 结构化 workflow ⇄ 规则引擎/DSL ⇄ 编译型代码`

- 关键词是 **⇄**（双向箭头）。组件可以向确定端移动（**支柱化**），也可以向统计端回退（**反向支柱化**）
- 移动方向的依据：该组件当前行为的确定性程度、稳定性要求、变更频率
- 同一系统中不同组件可以处于不同位置——这不是过渡态，混合共存本身就是稳态

### 2.2 支柱化：从统计到确定

- 定义：将反复验证的行为模式从谱系的统计端推向确定端，固化为确定性实现
- **[案例 B]**：Go Atlas behavior-analyzer——最初的行为分类可以用 LLM prompt 实现（统计端），但经过反复验证后，行为模式被证明是有限且确定的，于是固化为 tree-sitter AST 分析 + 规则匹配（确定端）。固化后：行为方差从不可预测降为零，执行时间从秒级降为毫秒级，不再需要 API 调用
- **[案例 C]**：test-issue-detector——测试质量判断从人工/LLM 审查固化为 4 种确定性检测规则（orphan test、zero assertion、debug test、large test）。固化前后的对比数据：检测延迟、一致性、误报率
- 每一步支柱化伴随**断言加固**——用测试锁定已固化的行为，防止回退时丢失保证

### 2.3 反向支柱化：从确定到统计

- 定义：当问题域变得更模糊、统计方法被证明优于规则时，将确定性实现回退到统计端
- 不是失败，是对问题域变化的正确响应
- **[案例 D]**：历史案例——规则驱动的垃圾邮件过滤器被 ML 分类器取代；手写 parser 被 LLM 语义解析取代。选择一个有具体数据的案例展开
- **[案例 E - 如果有]**：ArchGuard 自身是否有反向支柱化的实例？如果没有，诚实说明这个概念在本项目中尚未被检验

### 2.4 收敛谱系的权衡

| 维度 | 统计端（prompt/agent） | 确定端（编译型代码） |
|------|----------------------|-------------------|
| 编写成本 | 低（自然语言描述） | 高（完整实现） |
| 运行时信息传输 | 高（每次需传上下文） | 低（已编译） |
| 行为方差 | 高 | 极低 |
| 灵活性 | 极高 | 低 |
| 变更成本 | 极低 | 高 |

*注：上一版提纲中"残差描述长度"一列造成了术语混淆。此处拆分为"编写成本"和"运行时信息传输"两个独立维度，排序方向相反。*

### 2.5 晶体化：混合共存作为稳态

*这是对 "Software 2.0" 替代叙事的修正：不是 1.0 被 2.0 取代，而是两者共晶。*

- Karpathy 的 "Software 2.0" 暗示了一个从确定性代码到神经网络的**替代**方向。实际发生的不是替代，而是**分化**——同一系统中，部分组件沿谱系向确定端移动，部分向统计端移动，最终形成异构架构
- **晶体化**：系统演化为以确定性实现为骨架、统计推理为填充的混合架构。关键断言——**这不是过渡态，混合共存本身就是稳态**。不需要被"解决"为纯 1.0 或纯 2.0
- 这命名了一种已经广泛存在但尚未被显式认知的架构形态：你的系统一部分是编译型代码，一部分是 prompt/agent workflow——这是正常的、持久的终态，不是"还没来得及统一"的中间状态
- **[案例 F-1]**：ArchGuard 自身的晶体化——骨架：parser、archjson-mapper、mermaid-generator、query-engine（确定性代码，2787+ 测试守护）；填充：LLM grouping、MCP tools 的语义分析（统计推理接口）。两者通过 ArchJSON 中间表示解耦
- 晶体化的健康条件：骨架和填充之间有清晰的接口契约；骨架部分有充分的测试覆盖；填充部分的行为方差在可接受范围内

### 2.6 系统级相变

- 当多个组件同时面临反向支柱化压力时，骨架开始溶解——这不再是逐个组件的决策，而是系统级的相变信号
- 渐进策略：保持骨架，逐个替换填充
- 整体迁移：承认旧骨架不再适用，重建新骨架
- 与晶体化的关系：晶体化是稳态；相变是从一个稳态到另一个稳态的跃迁。触发条件是外部驱动（问题域整体变化、新技术范式），而非内部渐进演化
- **[案例 F-2]**：微服务硬编码编排 → agent-based 动态调度；或其他有具体数据的系统级迁移案例

## III. 操作判据：扩张与收缩

*从概念到可执行的 if-then 规则。这一节是给 LLM 工具开发者的。*

### 3.1 呼吸节律

- 扩张：吸收新需求、探索新方案、增加代码量
- 收缩：重构、提取共享结构、加固约束
- 健康的开发 = 有意识地交替两个相位
- 这发生在多个时间尺度上：
  - 微观（分钟级）：TDD 的 Red-Green-Refactor
  - 中观（天级）：一个 PR 内的功能开发 + 清理
  - 宏观（月级）：版本的功能扩张 + 架构收缩

### 3.2 收缩信号（何时从扩张切换到收缩）

可操作的检测条件：

```
IF 添加新功能需要触碰的文件数 > 近期均值的 2x
   → 模块边界可能已腐化，建议先重构

IF 测试失败从随机分布变为集群分布（多个失败集中在同一模块）
   → 结构性问题浮现，建议收缩

IF 同一模式在 3 处以上重复出现
   → 提取共享抽象的时机

IF 新加入的信息处理者（人或 LLM）理解模块 X 需要的前置概念数 > 5
   → 模块 X 的描述量过高，需要压缩
```

### 3.3 扩张信号（何时从收缩切换到扩张）

```
IF 继续重构的变更扇出 < 2 个文件
   → 重构收益饱和，停止

IF 外部需求队列非空且优先级高于内部压缩
   → 切换到扩张（资源约束优先于信息论判据）

IF 新增测试的 bug 发现率连续 N 轮为零
   → 约束空间边界已充分定义，停止加测试
```

### 3.4 支柱化/反向支柱化决策

```
向确定端移动（支柱化）的条件:
  - 行为模式已被充分测试验证
  - 稳定性/可审计性要求高
  - 行为变更频率低

向统计端回退（反向支柱化）的条件:
  - 问题域变得更模糊或输入分布漂移
  - 统计方法在准确性/灵活性上显著优于规则
  - 规则的边界情况数量持续增长

反模式:
  - 过早支柱化：行为尚未稳定就固化 → 频繁重写
  - 拒绝支柱化：行为已确定仍用 LLM → 浪费算力 + 引入不确定性
  - 拒绝反向支柱化：问题域已变，坚持用越来越复杂的规则 → 脆性系统
```

### 3.5 LLM agent 的操作协议

*这是本文与纯方法论文档的核心差异化。*

- LLM agent 自身处于收敛谱系的统计端。它的输出天然有行为方差
- 架构上下文（ArchJSON / Mermaid 图）的作用：缩小 agent 的搜索空间，提供约束空间的边界信息
- agent 在扩张模式下应被允许做什么：生成新代码、探索方案、增加文件
- agent 在收缩模式下应被限制什么：不应引入新抽象层，应优先重用已有结构
- agent 如何判断自己的输出应该被支柱化：如果同一个 prompt 模式被反复使用且结果稳定 → 固化为确定性代码
- **[案例 G]**：一个 Claude Code 在有/无架构上下文时做出不同决策的具体对比——同一个重构任务，同一个 prompt，架构上下文改变了什么

## IV. 反馈与渐进建设

*两个实践模式，不引入新术语。*

### 4.1 反馈机制

- 测试是约束空间边界的一种反馈，但不是唯一的
- 所有反馈统一为"对系统行为的信息采样"，区别在于：
  - 时间尺度：毫秒级（单元测试）→ 周/月级（A/B 实验）
  - 作用范围：函数级 → 系统级 → 用户旅程级
- 反馈前置原则：同一信息，获取越早价值越高（Shift Left 的信息论表述）
- 开发阶段与反馈配置：
  - 扩张阶段：偏向快速验证、宽松覆盖率
  - 收缩阶段：偏向精确测试、回归守护
- 测试投资优先指向**高敏感区**：变更频率高 × 耦合度高 × 修改后果大

### 4.2 能力的渐进建设

*以 ArchGuard test analysis 功能的演化为贯穿案例。*

- 从零建立任何工程能力（测试、监控、CI/CD）的渐进路径：
  1. **无形式化能力**：所有判断依赖人工/LLM 临时审查
  2. **低精度初始覆盖**：利用 LLM 或启发式规则提供粗粒度反馈（"有比没有好"）
  3. **高频判断固化**：将反复验证的判断固化为确定性实现——自动化测试、静态规则、监控告警
  4. **持续平衡**：确定性实现覆盖高敏感区，LLM/人工审查覆盖长尾。完全替代是理想化的——实际系统在阶段 3 和 4 之间振荡
- 这个过程本身就是沿收敛谱系的支柱化
- **[案例 H]**：ArchGuard test analysis 的具体演化路径——Plan 09（初始设计）→ Go 验证（meta-cc/lmdeploy）→ C++ 验证（llama.cpp）→ Java/Python 修复（Plan 40-49）。每个阶段的产出物、度量变化、哪些判断被固化

## V. 与现有方法论的差异分析

*不做礼貌映射。只回答：用了 GIM 视角，哪些决策会做出不同的选择？*

### 5.1 vs. 敏捷/XP

- 敏捷说"迭代"，但没给出"这次迭代应该扩张还是收缩"的判据
- GIM 的增量：§3.2/§3.3 的信号检测给出了相位切换的具体条件
- **差异化决策示例**：一个 Sprint 中 tech debt 修复被无限推迟——敏捷框架下这是 PO 的优先级决策；GIM 框架下，如果收缩信号持续触发，这是一个需要工程团队escalate的结构性问题，不是优先级问题

### 5.2 vs. Wardley Maps

- Wardley Maps 的演化轴（Genesis → Commodity）描述市场成熟度，**单向**移动
- 收敛谱系描述实现机制的确定性，**双向**移动
- 互补：Wardley 告诉你"build vs. buy"；GIM 告诉你"规则 vs. 模型"
- **差异化决策示例**：一个组件在 Wardley Maps 上已到 Commodity 阶段（应该买/用现成的），但在收敛谱系上仍需要反向支柱化（问题域变化导致现成方案不适用）——两个框架给出不同建议时如何决策

### 5.3 vs. LLM 工程（Context Engineering / Agent 框架）

- Context Engineering = 降低信息处理者理解系统所需的描述量
- GIM 的增量：收敛谱系将"代码系统"和"模型系统"统一为同一维度上的不同位置，而非需要不同工具链管理的两类对象
- **差异化决策示例**：团队在讨论"这个功能应该用 prompt 实现还是写代码实现"——传统框架下这是 ML vs. SWE 的工具选择；GIM 框架下这是收敛谱系上的位置决策，取决于行为确定性、稳定性需求和变更频率

## VII. 限制与诚实声明

### 7.1 GIM 不是什么

- 不是可量化验证的科学理论——Kolmogorov 复杂度不可计算，Fisher 信息在离散程序空间上缺乏严格定义
- 不提供精确的数值阈值——§3 的条件中的 "2x"、"3 处以上"、"5 个概念" 是启发式，不是经验定律
- 不替代资源规划——"应该重构"不等于"现在就重构"
- 不替代领域专家的判断

### 7.2 GIM 的失败模式

- 以"扩张阶段"为借口拒绝所有重构——收缩信号是硬约束，不是建议
- 以"压缩"合理化过度抽象——为不存在的共享结构创造抽象会增加描述量
- 用 GIM 术语包装常识——如果术语映射没有改变任何决策，它就是装饰
- 忽视成本约束——信息论判据和资源约束是两个独立维度，都需要满足

### 7.3 未验证的概念

- 反向支柱化：在 ArchGuard 项目中**尚无实例**。概念来自外部案例观察（垃圾邮件过滤器、手写 parser），未在本项目中直接检验
- 系统级相变（大规模反向支柱化）：纯理论构造，无第一手数据
- §3 的操作判据中的阈值：基于直觉设定，未经统计验证

### 7.4 系统边界

- 以共享部署单元或共享测试套件的范围为分析边界
- 跨边界按接口契约处理
- 边界选择本身是架构决策，没有完美答案
