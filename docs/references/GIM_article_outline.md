# 几何信息方法（GIM）：软件系统中的收敛谱系

**Geometric Information Method: The Convergence Spectrum in Software Systems**

状态：提纲 v2
日期：2026-03-31
目标受众：使用 LLM 工具的软件架构师、LLM 辅助开发工具的构建者

---

## Thesis Statement

> 在 LLM 与人类共同参与的软件开发中，每个组件都处于一条从统计推理到确定性代码的连续谱系上。支柱化和反向支柱化是这条谱系上的双向移动，晶体化是系统级的稳态。理解这一点，就能回答"这个功能该用 prompt 还是代码实现"这一类日常决策。

## 0. 开场：一个具体的观察

*不做理论辩护。从一个感性的、3 句话可以说完的事实开始。*

我们把 ArchGuard 的架构图喂给 Claude Code，让它重构一个 492 行的 God Class。它精确地按耦合度从低到高提取了 4 个模块——先提取零依赖的纯函数，最后处理最高耦合的组件。没有架构图时，它会试图一次性重写整个文件。

架构图不包含任何代码。它是信息的**损失性压缩**。一个丢掉了大量细节的表示，为什么反而提升了决策质量？

因为软件开发不是"写代码"，而是在一个巨大的可能性空间中**搜索**。压缩后的结构表示缩小了搜索范围，过滤掉了不相关的维度。这对人和 LLM 同样成立。而在 LLM 辅助开发的时代，还有一个更深的问题：系统中哪些部分应该由确定性代码实现，哪些应该交给 LLM？这两者如何共存？

## I. 铺垫：搜索、压缩与约束空间

*三个概念，用日常语言引入，不造新术语。它们是理解收敛谱系的前置知识。*

### 1.1 约束空间

- 所有能通过测试、满足类型检查、符合业务规则的实现，构成一个"约束空间"
- 开发 = 在这个空间中找到一个点；重构 = 在功能等价的前提下移动到一个更好的点
- "更好"的含义：向新加入者（人或 LLM）解释系统所需的概念数更少
- 关键不是"谁在搜索"，而是搜索的特征——人类搜索带宽低但方向感强；LLM 搜索带宽高但容易漂移出约束空间（"幻觉"）

### 1.2 压缩即理解

- 好的抽象 = 捕获真实的共享结构，使总描述量减少
- 过度抽象 = 抽象本身的复杂度超过它消除的重复，总描述量反增
- 这不是新洞察（奥卡姆剃刀、YAGNI），但它提供了一个**统一的判据**：任何工程决策都可以用"描述量是增还是减"来评估
- **描述量的代理观测**（不是精确数值，是可比较的序关系）：
  - 向新人/新 LLM 解释系统需要多少概念
  - 添加一个功能需要触碰多少文件
  - 存在多少"历史原因"的间接层
  - 移除一个组件后有多少其他组件需要修改
- **[案例 A 侧面：描述量过高的代理观测]**：DiagramProcessor 被触碰 34 次 commit，与 6+ 高 churn 文件共变，承担 6 种混合职责——每一个数字都是描述量过高的信号。ArchGuard 的 git churn × 耦合度分析将其变更风险评分定为 0.78（CRITICAL），直接触发了重构决策
  - **来源**：proposal commit `99f71d3`，实现 commit `429159d`，验证 commit `bf6c2c6` (2026-03-13 ~ 2026-03-30)
  - **执行**：基于耦合分析确定提取顺序（零依赖→纯函数→有状态→最高耦合），492→291 行，4 个新模块
  - **验证**：FIM 实验确认文件级 κ 改善（329.69→327.64），Mantel r=0.773 p=0.011

### 1.3 多目标性

- 同一系统存在多个正当的压缩目标：可理解性（语义命名、层级结构）vs. 执行效率（运行时性能、最小体积）
- 它们的最优解通常不重合：esbuild 的 minification 优化执行效率；架构工具优化可理解性
- 对 LLM 辅助开发的意义：LLM 消费的是**可理解性压缩**的产物，不是执行效率压缩的产物。架构图有用，minified 代码没用

## II. 核心贡献：收敛谱系

*这是 GIM 的核心原创概念。给它最多篇幅。*

### 2.1 谱系定义

- 系统中的每个组件都处于一条连续谱系上——

  `统计推理（LLM/prompt） ⇄ 结构化 workflow ⇄ 规则引擎/DSL ⇄ 编译型代码`

- 关键词是 **⇄**（双向箭头）。组件可以向确定端移动（**支柱化**），也可以向统计端回退（**反向支柱化**）
- 移动方向的依据：该组件当前行为的确定性程度、稳定性要求、变更频率
- 同一系统中不同组件可以处于不同位置——这不是过渡态，混合共存本身就是稳态

### 2.2 支柱化：从统计到确定

- 定义：将反复验证的行为模式从谱系的统计端推向确定端，固化为确定性实现
- **[案例 B]**：Go Atlas behavior-analyzer 支柱化
  - **来源**：Plan 16 proposal `docs/archive/refactoring/proposals/16-go-architecture-atlas.md`，实现 commit `9897d3a` (2026-02-24)，质量改进 commit `fbbfdcd` (2026-02-25)
  - **支柱化决策依据**：Proposal 中的"可恢复性真值表"量化了每种行为的确定性程度——Package imports: 100%, Interface usage: ~85%, Goroutine spawn: ~70%, HTTP entry: ~70%, Channel edges: <20%。只有确定性足够高的行为被固化为规则
  - **固化结果**：4 个 graph builder（PackageGraphBuilder、CapabilityGraphBuilder、GoroutineTopologyBuilder、FlowGraphBuilder），纯 tree-sitter AST 分析，~100 个测试守护
  - **未支柱化的边界**：Channel 通信（<20% 可恢复性）未固化——诚实承认静态分析的极限
  - **支柱化后特征**：行为方差=零，执行时间=毫秒级，无 API 依赖，7 轮 TDD 迭代精化规则
- **[案例 C]**：test-issue-detector 支柱化
  - **来源**：Plan 09 `docs/plans/plan-09-test-analysis.md`，实现 commit `68910ef` + `2395239` (2026-03-11)
  - **设计理念**："Pattern-First"——工具提供确定性基线检测，Claude Code 通过 `TestPatternConfig` 在上层修正。这本身就是收敛谱系的微观体现：骨架（4 条规则）确定性，填充（模式配置修正）交给统计推理
  - **4 条确定性规则**：zero_assertion（testCaseCount>0 且 assertionCount=0）、orphan_test（无覆盖链接）、skip_accumulation（skipCount/testCaseCount>0.2）、assertion_poverty（assertionDensity<1）
  - **架构决策**：Method C（两阶段协作）——插件提供原始测试结构（AST 解析），TestAnalyzer 做跨文件聚合（确定性模式匹配），明确拒绝了"插件内全包"和"独立 AST 重解析"两个替代方案
- 每一步支柱化伴随**断言加固**——用测试锁定已固化的行为，防止回退时丢失保证

### 2.3 反向支柱化：从确定到统计

- 定义：当问题域变得更模糊、统计方法被证明优于规则时，将确定性实现回退到统计端
- 不是失败，是对问题域变化的正确响应
- **[案例 E]**：ArchGuard Project Semantics Explorer——内部反向支柱化实例
  - **来源**：`src/analysis/project-semantics-explorer.ts` (168 行)，ADR-008 `docs/adr/008-llm-semantic-exploration-before-analysis.md`，Plan 51
  - **反向支柱化过程**：项目约定检测（非生产目录、barrel files、自定义断言模式、架构层识别）最初用硬编码默认值（确定端）。跨项目验证（meta-cc、lmdeploy、llama.cpp）暴露了规则无法覆盖的项目差异——ADR-008 显式记录"硬编码规则无法处理项目特定约定"。于是引入 Claude CLI 调用（统计端）进行语义探索
  - **反向支柱化后的架构**：三层 merge（用户配置 > LLM 结果 > 硬编码默认），保留确定性骨架（默认值 + 用户覆盖），增加统计填充（LLM 探索），结果通过 `.archguard/project-semantics.json` 缓存（目录树 hash 失效）
  - **补充——同一项目中的对称支柱化**：Mermaid LLM grouping（commit `d09d40f`, 2026-01-28）是**反方向**的运动——~2000 行 LLM 集成代码被删除，改为纯启发式分组，因为"proved unreliable and slow"。同一项目、两个组件、相反的谱系方向，恰好证明双向移动

### 2.4 收敛谱系的权衡

| 维度 | 统计端（prompt/agent） | 确定端（编译型代码） |
|------|----------------------|-------------------|
| 编写成本 | 低（自然语言描述） | 高（完整实现） |
| 运行时信息传输 | 高（每次需传上下文） | 低（已编译） |
| 行为方差 | 高 | 极低 |
| 灵活性 | 极高 | 低 |
| 变更成本 | 极低 | 高 |

*注：上一版提纲中"残差描述长度"一列造成了术语混淆。此处拆分为"编写成本"和"运行时信息传输"两个独立维度，排序方向相反。*

### 2.5 晶体化：混合共存作为稳态

*这是对 "Software 2.0" 替代叙事的修正：不是 1.0 被 2.0 取代，而是两者共晶。*

- Karpathy 的 "Software 2.0" 暗示了一个从确定性代码到神经网络的**替代**方向。实际发生的不是替代，而是**分化**——同一系统中，部分组件沿谱系向确定端移动，部分向统计端移动，最终形成异构架构
- **晶体化**：系统演化为以确定性实现为骨架、统计推理为填充的混合架构。关键断言——**这不是过渡态，混合共存本身就是稳态**。不需要被"解决"为纯 1.0 或纯 2.0
- 这命名了一种已经广泛存在但尚未被显式认知的架构形态：你的系统一部分是编译型代码，一部分是 prompt/agent workflow——这是正常的、持久的终态，不是"还没来得及统一"的中间状态
- **[案例 F-1]**：ArchGuard 自身的晶体化结构
  - **来源**：代码库现状 + commit `d09d40f`（LLM grouping 移除）+ `src/analysis/project-semantics-explorer.ts`（LLM 探索引入）
  - **确定性骨架**：语言插件（5 种语言，tree-sitter AST 解析）、Mermaid 生成器（启发式分组 + 5 层验证管线）、Git 历史分析（git-log-reader + history-aggregator）、测试分析（TestAnalyzer + TestIssueDetector，4 条规则）、查询引擎（确定性聚合）。3254+ 测试守护
  - **统计填充**：Project Semantics Explorer（Claude CLI 调用，`--explore`/`--no-explore` 开关控制）、MCP Tools 接口（15+ 工具供 Claude Code 查询，工具本身是确定性的，消费者是统计的）
  - **解耦机制**：骨架与填充通过 `ProjectSemantics` JSON 类型 + 三层 merge 解耦（用户配置 > LLM 结果 > 硬编码默认）
  - **演化证据**：LLM grouping 曾是填充（commit `d09d40f` 前），被支柱化为启发式规则；Project Semantics 曾是骨架（硬编码默认），被反向支柱化为 LLM 探索。系统整体保持晶体化稳态，但内部组件在谱系上持续移动
- 晶体化的健康条件：骨架和填充之间有清晰的接口契约；骨架部分有充分的测试覆盖；填充部分的行为方差在可接受范围内

### 2.6 系统级相变

- 当多个组件同时面临反向支柱化压力时，骨架开始溶解——这不再是逐个组件的决策，而是系统级的相变信号
- 渐进策略：保持骨架，逐个替换填充
- 整体迁移：承认旧骨架不再适用，重建新骨架
- 与晶体化的关系：晶体化是稳态；相变是从一个稳态到另一个稳态的跃迁。触发条件是外部驱动（问题域整体变化、新技术范式），而非内部渐进演化

## III. 操作判据：扩张与收缩

*从概念到可执行的 if-then 规则。这一节是给 LLM 工具开发者的。*

### 3.1 呼吸节律

- 扩张：吸收新需求、探索新方案、增加代码量
- 收缩：重构、提取共享结构、加固约束
- 健康的开发 = 有意识地交替两个相位
- 这发生在多个时间尺度上：
  - 微观（分钟级）：TDD 的 Red-Green-Refactor
  - 中观（天级）：一个 PR 内的功能开发 + 清理
  - 宏观（月级）：版本的功能扩张 + 架构收缩

### 3.2 收缩信号（何时从扩张切换到收缩）

可操作的检测条件：

```
IF 添加新功能需要触碰的文件数 > 近期均值的 2x
   → 模块边界可能已腐化，建议先重构

IF 测试失败从随机分布变为集群分布（多个失败集中在同一模块）
   → 结构性问题浮现，建议收缩

IF 同一模式在 3 处以上重复出现
   → 提取共享抽象的时机

IF 新加入的信息处理者（人或 LLM）理解模块 X 需要的前置概念数 > 5
   → 模块 X 的描述量过高，需要压缩
```

- **[案例 H 侧面：收缩信号触发]**：ArchGuard test analysis 初始实现后，自身 orphan 测试率为 94%（几乎全部误报）。这是一个集群性的质量信号——不是随机的个别失败，而是整个检测逻辑的系统性偏差。此信号在功能扩张期间（Plan 09 刚完成，多语言支持正在并行推进）强制触发了收缩：暂停新功能，优先修复检测精度。最终经过 Go/Python/C++/TS 四轮验证修复，orphan 率 94%→9.5%
  - **来源**：MEMORY.md 验证数据，Plans 40-49 commits (2026-03-13)

### 3.3 扩张信号（何时从收缩切换到扩张）

```
IF 继续重构的变更扇出 < 2 个文件
   → 重构收益饱和，停止

IF 外部需求队列非空且优先级高于内部压缩
   → 切换到扩张（资源约束优先于信息论判据）

IF 新增测试的 bug 发现率连续 N 轮为零
   → 约束空间边界已充分定义，停止加测试
```

- **[案例 H 侧面：扩张信号触发]**：test analysis 的 Plans 40-49 修复完成后，剩余 orphan 为合法边界情况（Go 无 struct 包、Python 纯函数模块）——新增规则的边际发现率趋近于零。此时切换回扩张：C++ 支持（Plan 24-26）、渲染性能优化（Plan 23）继续推进

### 3.4 支柱化/反向支柱化决策

```
向确定端移动（支柱化）的条件:
  - 行为模式已被充分测试验证
  - 稳定性/可审计性要求高
  - 行为变更频率低

向统计端回退（反向支柱化）的条件:
  - 问题域变得更模糊或输入分布漂移
  - 统计方法在准确性/灵活性上显著优于规则
  - 规则的边界情况数量持续增长

反模式:
  - 过早支柱化：行为尚未稳定就固化 → 频繁重写
  - 拒绝支柱化：行为已确定仍用 LLM → 浪费算力 + 引入不确定性
  - 拒绝反向支柱化：问题域已变，坚持用越来越复杂的规则 → 脆性系统
```

### 3.5 LLM agent 的操作协议

*这是本文与纯方法论文档的核心差异化。*

- LLM agent 自身处于收敛谱系的统计端。它的输出天然有行为方差
- 架构上下文（ArchJSON / Mermaid 图）的作用：缩小 agent 的搜索空间，提供约束空间的边界信息
- agent 在扩张模式下应被允许做什么：生成新代码、探索方案、增加文件
- agent 在收缩模式下应被限制什么：不应引入新抽象层，应优先重用已有结构
- agent 如何判断自己的输出应该被支柱化：如果同一个 prompt 模式被反复使用且结果稳定 → 固化为确定性代码
- **[案例 G]**：*待补充——一个 Claude Code 在有/无架构上下文时做出不同决策的具体对比。A/B/C 实验（`docs/experiments/git-methodology-ab-test.md`）已设计未执行。案例 A 提供了间接证据（架构分析驱动的重构决策优于无上下文的重写），但尚缺直接的控制实验*

## IV. 反馈与渐进建设

*两个实践模式，不引入新术语。*

### 4.1 反馈机制

- 测试是约束空间边界的一种反馈，但不是唯一的
- 所有反馈统一为"对系统行为的信息采样"，区别在于：
  - 时间尺度：毫秒级（单元测试）→ 周/月级（A/B 实验）
  - 作用范围：函数级 → 系统级 → 用户旅程级
- 反馈前置原则：同一信息，获取越早价值越高（Shift Left 的信息论表述）
- 开发阶段与反馈配置：
  - 扩张阶段：偏向快速验证、宽松覆盖率
  - 收缩阶段：偏向精确测试、回归守护
- 测试投资优先指向**高敏感区**：变更频率高 × 耦合度高 × 修改后果大

### 4.2 能力的渐进建设

*以 ArchGuard test analysis 功能的演化为贯穿案例。*

- 从零建立任何工程能力（测试、监控、CI/CD）的渐进路径：
  1. **无形式化能力**：所有判断依赖人工/LLM 临时审查
  2. **低精度初始覆盖**：利用 LLM 或启发式规则提供粗粒度反馈（"有比没有好"）
  3. **高频判断固化**：将反复验证的判断固化为确定性实现——自动化测试、静态规则、监控告警
  4. **持续平衡**：确定性实现覆盖高敏感区，LLM/人工审查覆盖长尾。完全替代是理想化的——实际系统在阶段 3 和 4 之间振荡
- 这个过程本身就是沿收敛谱系的支柱化
- **[案例 H]**：ArchGuard test analysis 渐进建设全过程
  - **来源**：Plan 09 commit `68910ef`，多语言实现 commits `2395239`/`b2f742e`/`c5d35e2`/`f1290ec` (2026-03-11)，精度修复 Plans 40-49 commits (2026-03-13)，MEMORY.md 验证数据

## V. 与现有方法论的差异分析

*不做礼貌映射。只回答：用了 GIM 视角，哪些决策会做出不同的选择？*

### 5.1 vs. 敏捷/XP

- 敏捷说"迭代"，但没给出"这次迭代应该扩张还是收缩"的判据
- GIM 的增量：§3.2/§3.3 的信号检测给出了相位切换的具体条件
- **实际决策案例**：ArchGuard 的 test analysis 精度修复（Plans 40-49）发生在功能扩张期间——C++ 支持、渲染优化等新功能并行推进。敏捷框架下，94% orphan 率的修复是一个可以排优先级的 tech debt item。GIM 框架下，集群性质量信号（§3.2）是硬约束——它不是 PO 的优先级决策，而是工程团队必须响应的结构性问题。实际结果：修复被优先执行，orphan 率 94%→9.5%，之后才继续扩张

### 5.2 vs. Wardley Maps

- Wardley Maps 的演化轴（Genesis → Commodity）描述市场成熟度，**单向**移动
- 收敛谱系描述实现机制的确定性，**双向**移动
- 互补：Wardley 告诉你"build vs. buy"；GIM 告诉你"规则 vs. 模型"
- **[待补充实际决策案例]**

### 5.3 vs. LLM 工程（Context Engineering / Agent 框架）

- Context Engineering = 降低信息处理者理解系统所需的描述量
- GIM 的增量：收敛谱系将"代码系统"和"模型系统"统一为同一维度上的不同位置，而非需要不同工具链管理的两类对象
- **实际决策案例**：ArchGuard 对两个组件做出了相反的"prompt vs. code"决策——Mermaid grouping 从 LLM 移向启发式规则（commit `d09d40f`，因为"unreliable and slow"），Project Semantics 从硬编码移向 LLM 探索（Plan 51，因为"硬编码规则无法处理项目特定约定"）。传统框架缺少统一判据解释为什么两个相反决策都是对的；收敛谱系可以——前者行为已确定且稳定性要求高，后者问题域模糊且输入分布差异大

## VII. 限制与诚实声明

### 7.1 GIM 不是什么

- 不是可量化验证的科学理论——Kolmogorov 复杂度不可计算，Fisher 信息在离散程序空间上缺乏严格定义
- 不提供精确的数值阈值——§3 的条件中的 "2x"、"3 处以上"、"5 个概念" 是启发式，不是经验定律
- 不替代资源规划——"应该重构"不等于"现在就重构"
- 不替代领域专家的判断

### 7.2 GIM 的失败模式

- 以"扩张阶段"为借口拒绝所有重构——收缩信号是硬约束，不是建议
- 以"压缩"合理化过度抽象——为不存在的共享结构创造抽象会增加描述量
- 用 GIM 术语包装常识——如果术语映射没有改变任何决策，它就是装饰
- 忽视成本约束——信息论判据和资源约束是两个独立维度，都需要满足

### 7.3 未验证的概念

- ~~反向支柱化：在 ArchGuard 项目中尚无实例~~ → **已确认**：Project Semantics Explorer（案例 E）。但样本量仍为 1，不足以泛化
- 系统级相变（大规模反向支柱化）：纯理论构造，无第一手数据。ArchGuard 内部的单组件反向支柱化不构成"系统级"证据
- §3 的操作判据中的阈值：基于直觉设定，未经统计验证
- A/B/C 实验（GIT 框架对 LLM 决策质量的影响）：已设计（`docs/experiments/git-methodology-ab-test.md`），尚未执行

### 7.4 系统边界

- 以共享部署单元或共享测试套件的范围为分析边界
- 跨边界按接口契约处理
- 边界选择本身是架构决策，没有完美答案
