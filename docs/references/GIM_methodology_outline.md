## 0. 前言：从理论到方法论的转向

- GIT 文献回顾与定位反思
- 为什么不追求量化验证：Kolmogorov 复杂度不可计算、Fisher 信息在离散程序空间上缺乏严格定义
- 方法论的合法性：UML/设计模式/敏捷的先例——定性认知工具在工程界的价值
- GIM 的定位：一组概念词汇 + 一套实践启发式 + 一个决策框架

## I. 核心概念词汇（Vocabulary）

*四个概念，不作为公理推导，而作为"思维透镜"引入。*

### 1.1 压缩即理解

- **思维模型**：任何信息系统可分解为生成规则 G（共享抽象：库、模式、框架）、残差 R（在 G 基础上的增量描述）、边界约束 A（测试、类型、契约）。这是思维工具，不是数学对象
- **好的抽象** = 捕获真实共享结构，使 G + R 的总量最小；**过度抽象** = G 本身膨胀，总量反增。与奥卡姆剃刀和 YAGNI 原则同源
- **多目标性**：同一系统存在多个正当的压缩目标——可理解性（语义命名、模块边界、层级结构）和执行效率（运行时性能、最小字节数）——它们的最优解通常不重合。esbuild 的 minification 优化执行效率；架构工具优化可理解性。两者在同一功能等价约束下操作
- **描述长度的操作定义**：在 GIM 中，"描述长度"不是一个可精确计算的数值，而是一个可比较的序关系。实践中的代理观测量包括：
  - 向一个新加入的信息处理者（人或 LLM）解释系统结构所需的概念数
  - 实现一个新功能需要触碰的文件/模块数
  - 存在多少"只是因为历史原因"才存在的间接层
  - 移除一个组件后，有多少其他组件需要修改

### 1.2 程序空间与可行区域

- 所有满足测试/约束的实现构成"可行区域"（放弃"流形"的严格拓扑含义）
- 开发 = 在可行区域内导航；重构 = 在功能等价的前提下压缩描述长度
- 所有参与开发的信息处理过程（无论由人、LLM 还是自动化工具执行）都是这个空间中的搜索器。软件开发涉及四种基本操作——搜索（探索可行区域）、压缩（提取共享结构）、约束（定义边界）、验证（测试/断言）——它们可以由任何信息处理者执行，关键不是"谁做"而是操作的特征

### 1.3 呼吸节律：扩张与收缩

- 扩张：吸收新需求、探索新方案、增加代码量
- 收缩：重构、提取库、压缩残差、加固断言
- 健康的开发 = 有意识地交替两个相位，而非一直扩张
- 相位切换的具体信号见 §2.1 和 §2.2

### 1.4 支柱化与晶体化：混合系统的演化模式

*呼吸节律（§1.3）描述时间维度——何时扩张、何时收缩。支柱化与晶体化描述空间维度——系统内部的异构结构如何形成。两者正交。*

- **收敛谱系**：系统组件沿以下谱系移动——
  `统计推理（LLM / 模型） ⇄ 结构化 workflow ⇄ 规则引擎/DSL ⇄ 编译型代码`
  谱系的两端不对应"谁在操作"，而对应行为的确定性程度。**移动是双向的**——支柱化向确定端推进，但当问题域变化、输入变得更模糊、或统计方法被证明优于规则时，组件也可以从确定端回退到统计端（反向支柱化 / de-pillarization）。例如：规则驱动的垃圾邮件过滤器被 ML 分类器取代；手写 parser 被 LLM 语义解析取代
- **收敛谱系各层级的权衡**：

  | 实现层级 | 残差描述长度 | 行为方差 | 灵活性 | 变更成本 |
  |----------|------------|---------|--------|---------|
  | Prompt / Agent | 极低 | 较高 | 极高 | 极低 |
  | 结构化 Workflow | 低 | 中等 | 高 | 低 |
  | 专用模型 | 中等 | 中等偏低 | 中等 | 中等 |
  | DSL / 规则引擎 | 中高 | 低 | 中低 | 中等 |
  | 编译型代码 | 高 | 极低 | 低 | 高 |

  选择层级的依据是：该组件当前的行为确定性、稳定性要求、变更频率。同一系统中不同组件可以处于不同层级
- **支柱化（Pillarization）**：收敛过程中的局部操作——将行为模式从谱系的统计端推向确定端。固化后的实现成为"结构性支柱"。每一步伴随断言加固（用测试锁定已固化的行为）
- **反向支柱化（De-pillarization）**：将已确定的实现回退到统计端。当问题域复杂度超出规则可表达的范围、或统计方法的准确性/灵活性显著优于确定性实现时，这是合理的收敛方向
- **晶体化（Crystallization）**：支柱化的累积效应在系统层面形成的宏观稳态。系统演化为以确定性实现为骨架、统计推理为填充的异构架构。这不是过渡态——混合共存本身就是稳态
- **熔融（Melting）**：反向支柱化的累积效应在系统层面形成的宏观相变。当系统中大量确定性组件被统计推理取代时，骨架溶解、填充比例反转。与晶体化对称，但特征不同——熔融通常由外部驱动（问题域整体变化、新技术范式出现），速度更快，破坏性更大。典型例子：规则引擎驱动的推荐系统被整体替换为 ML pipeline；手写 parser + formatter 被 LLM 语义处理整体取代；微服务硬编码编排迁移到 agent-based 动态调度

## II. 实践启发式（Heuristics）

*从概念词汇导出的操作指南*

### 2.1 扩张阶段的纪律

- 新功能引入时，同步定义边界约束（测试/类型/契约）
- 高带宽搜索器的扩张陷阱：高速信息处理过程（无论人还是 LLM）倾向于生成新代码而非发现和重用已有抽象
- **收缩信号**（出现以下情况时，应从扩张切换到收缩）：
  - 新增功能的边际描述长度增速加快（保持一致性越来越难）
  - 行为方差超过可接受阈值（系统变得不可预测）
  - 测试失败从随机分布变为集群分布（结构性问题浮现）
  - 添加新功能时需要触碰越来越多的文件

### 2.2 收缩阶段的策略

- 识别重复模式 → 提取为共享生成规则（库/框架/DSL）
- 断言加固：在高敏感区（修改后果大、变更频率高的代码区域）增加不变量约束
- 描述长度审计：向系统新加入者解释结构需要多少概念？存在多少"历史遗留"的间接层？代码是否可以用更少的概念表达相同功能？
- 支柱化决策：对于行为已确定且稳定性要求高的组件，沿收敛谱系向确定端推进
- 反向支柱化决策：对于问题域已变化或统计方法已证明更优的组件，允许回退到统计端
- 晶体化/熔融审视：定期评估系统整体的骨架/填充比例。晶体化方向——确认混合结构是否与各组件的成熟度匹配；熔融方向——识别是否出现大规模反向支柱化的信号（多个确定性组件同时面临问题域漂移），若是，则需要系统级的迁移策略而非逐个替换
- **扩张信号**（出现以下情况时，应从收缩切换到扩张）：
  - 继续压缩的边际收益趋近于零（重构收益递减）
  - 外部需求队列非空且优先级高于内部压缩
  - 新增测试的边际 bug 发现率骤降（可行区域边界已充分约束）

### 2.3 反馈机制的分类与配置

*测试只是约束可行区域边界的多种反馈机制之一。GIM 将所有反馈统一为"对系统行为的信息采样"。*

- **反馈的四要素**：每种反馈机制可分解为——
  - **探测**（如何提取信号）：执行+断言、指标采集、语义分析、用户行为埋点
  - **解码**（如何转化为可操作信息）：二值判断、统计异常检测、多值建议
  - **作用空间**（能触及的程序维度）：函数级、模块级、系统级、用户旅程级
  - **时间尺度**（从信号发射到信息可用的延迟）：毫秒级（单元测试）→ 周/月级（A/B 实验）
- **反馈前置原则**（Shift Left 的信息论表述）：同一信息，获取得越早，价值越高（修复成本随发现时间非线性增长）。在信息价值与获取成本的比值最大化约束下，优先部署时间尺度最小的反馈
- **反馈的边际收益递减**：每增加一种反馈机制，其边际信息增益递减——因为它不仅有自身的维护成本，还需要在已有反馈覆盖的空间中寻找残余信息
- **按开发阶段调整反馈配置**：
  - 扩张阶段：偏向高探索性反馈（原型测试、快速验证、宽松覆盖率容忍）
  - 收缩阶段：偏向高精度反馈（单元测试、变异测试、回归守护、细粒度监控）

### 2.4 测试投资的启发式

- 测试 = 约束可行区域的边界；高价值测试 = 排除大片不可行区域的测试
- 何时停止写测试：新增测试的边际信息增益递减时
- **TDD 作为反馈前置的实践**：根据反馈前置原则（§2.3），测试先于实现编写是信息论上的最优策略——在可行区域边界尚未确定时就预先约束，而非事后修补。越来越多的开发实践已经验证了这一点
- **迟滞效应**：即使采用 TDD，随着系统复杂度增长，仍然会持续发现既有实现中的缺陷或测试未覆盖的维度。这不是 TDD 的失败，而是信息论的结构性约束——在编写测试时，编写者的信息充分度低于系统运行后的信息充分度。迟滞是复杂度增长的必然伴生现象，而非可以消除的缺陷
- 测试投资优先指向**高敏感区**——修改后果大、变更频率高、耦合度高的代码区域

### 2.5 架构健康的代理信号

- 腐化信号：添加新功能时需要触碰越来越多的文件；循环依赖增加；新加入者的上手时间持续增长
- 压缩信号：多个功能可以通过同一抽象实现；模块可以独立理解和修改
- **代理度量**（非精确值，但可比较和追踪趋势）：
  - 实体数/关系数：结构规模
  - 强连通分量/循环数：结构纠缠程度
  - 变更扇出（一次修改触碰的文件数）：模块独立性
  - git churn × 耦合度：高敏感区定位

### 2.6 能力自举的阶段模型

*从零建立任何工程能力（测试、监控、文档、CI/CD）的渐进模式。与支柱化概念直接关联——自举过程本身就是一个沿收敛谱系的移动。*

- **阶段 0 — 初始状态（⊥）**：无形式化能力。所有判断依赖隐性知识和即席检查。不确定性最大
- **阶段 1 — 临时锚定**：利用通用信息处理能力（LLM 审视、经验规则、启发式检查）提供低精度但低成本的初始覆盖。不追求准确，追求"有总比没有好"
- **阶段 2 — 跨层下推（固化）**：将阶段 1 中反复出现的高价值判断固化为确定性实现——自动化测试、静态规则、监控告警。下推规则：
  - 高频触发优先：被反复确认的判断最先固化
  - 失败模式固化：每个发现的缺陷修复后，伴随一个精确的回归约束
  - 精度梯度上升：从模糊约束（"不应崩溃"）逐步精化为精确约束（"输出应在 [a, b] 区间内"）
- **阶段 3 — 代谢完成**：临时锚定层（高成本、高方差）被确定性实现（低成本、低方差）取代。系统进入自持运行
- **自举完成的信号**：新增覆盖的边际增益饱和；临时锚定层发现新问题的频率持续下降；确定性实现已覆盖所有高敏感区

## III. 决策框架（Decision Framework）

*面对具体工程决策时的 GIM 检查清单*

### 3.1 "该不该抽象？"

- 检查：这个模式是否在至少三处重复？
- 检查：抽象后的 G 增量是否小于它消除的 R？（操作判据：引入抽象后，向新加入者解释系统需要的概念数是增加了还是减少了？）
- 反模式：为假想的未来需求预建抽象

### 3.2 "该不该重构？"

- 检查：描述长度审计（§2.2）是否揭示了显著的冗余或不必要的间接层？
- 检查：§2.5 中的腐化信号是否持续出现？
- 检查：是否处于收缩相位？（扩张阶段不宜大规模重构）
- **成本约束**：重构决策首先是一个资源问题——预算、时间、团队负荷。GIM 提供的是"该不该"的信息论判据，而非"能不能"的资源判据。两者都需要满足

### 3.3 "该不该加测试？"

- 检查：目标区域是否属于高敏感区？（§2.5 的代理度量可辅助判断）
- 检查：现有反馈机制（§2.3）是否已充分约束该区域？
- 检查：新测试的维护成本是否可接受？

### 3.4 "该沿收敛谱系向哪个方向移动？"

- **向确定端（支柱化）的条件**：行为模式已被充分理解；稳定性/审计性要求高；行为变更频率低
- **向统计端（反向支柱化）的条件**：问题域变得更模糊或更复杂；统计方法在准确性/灵活性上显著优于规则；输入分布发生了规则无法覆盖的漂移
- **系统级判断——晶体化还是熔融？**：当多个组件同时面临反向支柱化压力时，这不再是逐个组件的决策，而是系统级的相变信号。此时需要评估：是渐进替换（保持骨架，逐个替换填充）还是整体迁移（承认旧骨架不再适用）
- **反模式**：
  - 过早支柱化：行为尚未稳定时就固化为代码，导致频繁重写
  - 拒绝支柱化：行为已确定，仍依赖统计推理，浪费算力且引入不确定性
  - 拒绝反向支柱化：问题域已变化，仍坚持用越来越复杂的规则覆盖边界情况，而非承认统计方法更适合
  - 无意识熔融：多个组件被逐个替换为统计实现，但没有意识到系统整体正在经历相变，缺乏统一的迁移策略和新的约束体系

## IV. 与现有方法论的关系

*GIM 不替代现有方法论。以下逐一说明 GIM 提供的增量视角——如果增量为零，则诚实承认。*

### 4.1 GIM 与敏捷/XP

- **术语映射**：Sprint ≈ 扩张-收缩周期；TDD 的 Red-Green-Refactor ≈ 微观呼吸节律；技术债 ≈ 累积的描述长度偏差
- **GIM 的增量贡献**：敏捷提供了"迭代"的节奏，但没有给出"何时停止重构转入新功能"的判据。§2.1/§2.2 的相位切换信号尝试填补这个缺口。此外，能力自举模型（§2.6）为敏捷团队中"测试后补"的常见困境提供了渐进路径

### 4.2 GIM 与 DDD（领域驱动设计）

- **术语映射**：Bounded Context ≈ 可行区域的独立子区域；Ubiquitous Language ≈ 可理解性压缩的优化
- **GIM 的增量贡献**：有限。DDD 已经是关于"如何划分和命名"的成熟方法论。GIM 的收敛谱系为 DDD 增加了一个维度——同一个 Bounded Context 内部的不同组件可以处于谱系的不同位置（晶体化），但这在 DDD 的实践中已经是常识，只是没有专门命名

### 4.3 GIM 与 Wardley Maps

- **结构相似性**：Wardley Maps 的演化轴（Genesis → Custom → Product → Commodity）与收敛谱系结构相似——都描述组件沿确定性/成熟度轴的移动
- **关键区别**：
  - Wardley Maps 的演化轴描述市场/生态位的成熟度，适用于战略决策（build vs buy vs outsource）
  - GIM 的收敛谱系描述实现机制的确定性，适用于技术决策（用规则还是用模型？该固化为代码还是保留为 prompt？）
  - Wardley Maps 的移动是单向的（Genesis → Commodity）；GIM 的移动是双向的（支柱化和反向支柱化）
- **互补性**：Wardley Maps 告诉你"这个组件在市场演化中处于什么位置"；GIM 告诉你"这个组件在技术实现中应该用什么确定性层级"。两者可以叠加使用

### 4.4 GIM 与架构决策记录（ADR）

- ADR = 对可行区域导航决策的显式记录
- 架构适应度函数 = 边界约束 A 的自动化实现

### 4.5 GIM 与 LLM 工程（MLOps / LLMOps / Agent 框架）

- Context Engineering = 降低信息处理者理解系统所需的描述长度
- Prompt = 可理解性压缩的极端形式
- "幻觉" = 搜索结果偏离可行区域（任何信息处理过程都可能产生，统计推理过程更频繁）
- **GIM 的增量贡献**：传统方法论将"代码系统"和"模型系统"视为需要不同工具链管理的两类对象。GIM 的收敛谱系将它们统一为同一维度上的不同位置，晶体化概念描述它们在同一系统内的共存稳态

## V. 工具需求

*GIM 方法论对工具的能力要求，不绑定到特定实现。*

### 5.1 架构压缩能力

- 将源代码转换为结构化中间表示（如 ArchJSON），沿可理解性方向压缩
- 提供多粒度的压缩投影（包/类/方法或语言特定的层级视图）
- 输出应可被任何信息处理者消费（人可读的图、机器可读的 JSON）

### 5.2 架构度量能力

- 高敏感区定位：git churn × 耦合度
- 结构纠缠度量：循环依赖数、强连通分量
- 描述长度代理：变更扇出（一次修改触碰的文件数）、模块独立性评分
- 趋势追踪：以上度量随时间的变化，而非单一快照

### 5.3 上下文生成能力

- 为任何参与开发的信息处理过程生成可消费的架构摘要
- 在代码修改前提供可行区域边界提示（"这次修改会影响哪些模块？"）
- 变更风险评估

## VI. 限制与诚实声明

*方法论的边界*

### 6.1 GIM 不是什么

- 不是可量化验证的科学理论
- 不提供精确的数值阈值（"何时重构"没有公式）
- 不替代领域专家的判断
- 不替代资源规划——GIM 提供"该不该"的信息论判据，不回答"能不能"的预算/时间/人力问题

### 6.2 GIM 的失败模式

*方法论被误用时的典型表现：*

- **以"还在扩张阶段"为借口拒绝所有重构**——呼吸节律的两个相位不是自由选择，收缩信号（§2.1）是硬约束
- **以"描述长度优化"合理化过度抽象**——压缩即理解的前提是"捕获真实共享结构"，为不存在的共享结构创造抽象会增加而非减少描述长度
- **用 GIM 术语包装已有常识**——如果将"Sprint = 扩张-收缩周期"说出口后没有改变任何决策，这个映射就是装饰性的，不应被当作洞察
- **忽视成本约束**——信息论上"应该重构"不等于"现在就重构"。资源约束是独立于 GIM 的硬边界

### 6.3 系统边界问题

- GIM 的"可行区域"以什么为边界？一个服务？一个 monorepo？一个平台？
- 实践建议：以共享部署单元或共享测试套件的范围为 GIM 分析的边界。跨边界的关系按接口契约处理，不按内部结构分析
- 这个问题没有完美答案——边界选择本身就是一个架构决策

### 6.4 已知的认知偏差风险

- "锤子效应"：不是所有工程问题都适合几何/信息论隐喻
- 过度抽象的诱惑：GIM 本身也受"G 膨胀"约束——本文档引入的术语数量应当是最小必要集
- 术语壁垒：如果团队成员不理解 GIM 术语，使用 GIM 反而增加沟通成本。方法论的价值在于改变决策，不在于统一词汇

### 6.5 与 GIT 理论文献的关系

- GIT 文献（本文档目录下的历史版本）提供了数学灵感来源
- GIM 保留了 GIT 的核心直觉，但放弃了其"精确科学"的定位声明
- 未来如果出现可行的量化方法，GIM 可以回归理论框架
