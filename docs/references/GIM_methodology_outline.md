## I. 核心概念词汇（Vocabulary）

*不作为公理推导，而作为"思维透镜"引入*

### 1.1 压缩即理解

- 生成规则 G / 残差 R / 边界约束 A 作为思维模型（非数学对象）
- 实践含义：好的抽象 = 捕获真实共享结构；过度抽象 = G 本身膨胀
- 与"奥卡姆剃刀"和 YAGNI 原则的关系

### 1.2 程序空间与可行区域

- 所有满足测试/约束的实现构成"可行区域"（放弃"流形"的严格拓扑含义）
- 开发 = 在可行区域内导航；重构 = 在功能等价的前提下压缩描述长度
- AI Agent 作为这个空间中的"近似搜索器"

### 1.3 呼吸节律：扩张与收缩

- 扩张：吸收新需求、探索新方案、增加代码量
- 收缩：重构、提取库、压缩残差、加固断言
- 健康的开发 = 有意识地交替两个相位，而非一直扩张

### 1.4 信息敏感度（Fisher 隐喻）

- 不同代码区域对系统行为的影响程度不同
- "高敏感区"= 修改后果大、测试价值高的代码
- 实践启发：优先为高敏感区编写测试和断言

### 1.5 认知压缩 vs 机器压缩

- 同一系统对人和机器有不同的"最优表示"
- 人类需要语义命名、模块边界、层级结构（认知 MDL）
- 机器需要执行效率、最小字节数（机器 MDL）
- 架构工具（如 ArchGuard）的角色：为人和 AI 提供不同的压缩视图

### 1.6 支柱化与晶体化：混合系统的演化模式

*呼吸节律（§1.3）描述时间维度——何时扩张、何时收缩。支柱化与晶体化描述空间维度——系统内部的异构结构如何形成。两者正交。*

- **收敛谱系**：在人-AI 协作开发中，系统组件沿以下谱系收敛——
  `通用 LLM + prompt → 结构化 workflow → 规则引擎/DSL → 编译型代码`
  每一级收敛都是用更确定、更低描述长度的机制替换更模糊、更高描述长度的机制，同时保持行为不变（测试持续通过）
- **支柱化（Pillarization）**：收敛过程中的局部操作。当某个 AI 生成的行为模式被充分理解且稳定性要求高时，将其固化为确定性代码。固化后的代码成为系统的"结构性支柱"——库函数、标准 API、确定性规则引擎。支柱化的每一步都伴随断言加固（用测试锁定已固化的行为）
- **晶体化（Crystallization）**：支柱化的累积效应在系统层面形成的宏观结构。系统整体演化为以确定性代码为骨架（晶格）、AI 能力为填充的混合架构。不同组件处于收敛谱系的不同位置——这不是过渡态，而是稳态
- **晶体化的关键洞察**：
  - 混合系统的稳态既不是"全代码"也不是"全 AI"，而是异构共存
  - 骨架（代码）提供可靠性和可审计性；填充（AI）提供灵活性和探索能力
  - 哪些组件成为骨架、哪些保留为填充，取决于该组件的行为确定性和稳定性要求
  - 传统方法论缺少对这种异构稳态的描述语言——DDD 描述代码结构，MLOps 描述模型运维，但没有方法论统一描述"组件 A 是确定性代码、组件 B 是 LLM prompt、组件 C 是规则引擎"的系统
- **ArchGuard 作为实证**：ArchGuard 自身的演化历史展示了晶体化过程——
  - 骨架：parser、archjson-mapper、mermaid-generator、query-engine（确定性代码）
  - 填充：LLM grouping、MCP tools for AI agents（AI 能力）
  - 支柱化实例：Go Atlas 的 behavior-analyzer 将原本可能需要 LLM 判断的"行为分类"固化为 tree-sitter 分析 + 规则匹配；test-issue-detector 将原本依赖人工审查的测试质量判断固化为确定性检测规则

## II. 实践启发式（Heuristics）

*从概念词汇导出的操作指南*

### 2.1 扩张阶段的纪律

- 新功能引入时，同步定义边界约束（测试/类型/契约）
- 避免"只扩不收"：设定收缩触发条件（如每 N 个功能后强制重构）
- AI Agent 的扩张陷阱：LLM 倾向于生成新代码而非重用已有抽象

### 2.2 收缩阶段的策略

- 识别重复模式 → 提取为共享生成规则（库/框架/DSL）
- 断言加固：在高敏感区增加不变量约束
- 描述长度审计：代码是否可以用更少的概念表达相同功能？
- 支柱化决策：对于行为已确定且稳定性要求高的组件，沿收敛谱系向代码端推进
- 晶体化审视：定期评估系统整体的骨架/填充比例，确认混合结构是否与各组件的成熟度匹配

### 2.3 测试投资的信息论启发

- 测试 = 约束可行区域的边界
- 高价值测试 = 排除大片不可行区域的测试
- 何时停止写测试：新增测试的边际信息增益递减时
- 测试与开发的相位差是正常的（测试滞后于开发扩张）

### 2.4 架构健康的代理信号

- 本征维度隐喻：系统的"真实自由度"应远小于代码量
- 腐化信号：添加新功能时需要触碰越来越多的文件
- 压缩信号：多个功能可以通过同一抽象实现 → 健康
- 循环依赖 = 拓扑缺陷（GIM 词汇中的"流形奇异点"）

### 2.5 人-AI 协作的节奏指导

- AI 作为"快速搜索器"：擅长扩张，需要人类引导收缩
- 人类作为"几何约束器"：定义架构边界、审查 AI 的抽象选择
- 工具作为"压缩中介"：ArchGuard 等工具提供机器可读的架构上下文

## III. 决策框架（Decision Framework）

*面对具体工程决策时的 GIM 检查清单*

### 3.1 "该不该抽象？"

- 检查：这个模式是否在至少三处重复？
- 检查：抽象后的 G 增量是否小于它消除的 R？
- 反模式：为假想的未来需求预建抽象

### 3.2 "该不该重构？"

- 检查：当前描述长度是否显著偏离可行区域的"测地线"（最短路径）
- 检查：添加新功能是否变得越来越困难（本征维度增长信号）
- 检查：是否处于收缩相位？（扩张阶段不宜大规模重构）

### 3.3 "该不该加测试？"

- 检查：目标区域的敏感度如何？（高变更频率 / 高耦合度）
- 检查：现有测试是否已充分约束该区域的可行空间？
- 检查：新测试的维护成本是否可接受？

### 3.4 "该支柱化还是保留为 AI 填充？"

- 检查：该组件的行为模式是否已被充分理解？（模糊 → 保留 AI；确定 → 支柱化）
- 检查：稳定性要求多高？（高可靠性 / 高审计性 → 支柱化；探索性 / 快速迭代 → 保留 AI）
- 检查：行为变更频率如何？（频繁变更 → AI 填充成本更低；稳定 → 代码维护成本更低）
- 反模式：过早支柱化（行为尚未稳定时就固化为代码，导致频繁重写）
- 反模式：拒绝支柱化（明明行为已确定，仍依赖 AI 推理，浪费算力且引入不确定性）

### 3.5 "该让 AI 做还是人做？"

- AI 适合：搜索空间大、约束明确、可验证的任务（扩张 + 有边界的探索）
- 人类适合：定义约束、选择抽象方向、判断认知 MDL 的任务（收缩 + 架构决策）
- 协作模式：人定义可行区域边界 → AI 在区域内搜索 → 人审查压缩质量

## IV. 与现有方法论的关系

*GIM 不替代，而是提供一个统一的"几何视角"*

### 4.1 GIM 与敏捷/XP

- Sprint = 一个扩张-收缩周期
- TDD 的 Red-Green-Refactor = 微观呼吸节律
- 技术债 = 累积的描述长度偏差

### 4.2 GIM 与 DDD（领域驱动设计）

- Bounded Context = 可行区域的子流形
- Ubiquitous Language = 认知 MDL 的优化

### 4.3 GIM 与架构决策记录（ADR）

- ADR = 对可行区域导航决策的显式记录
- 架构适应度函数 = 边界约束 A 的自动化实现

### 4.4 GIM 与 AI 辅助开发（Copilot / Claude Code / Codex）

- Context Engineering = 为 AI 提供压缩后的架构上下文
- Prompt = 认知 MDL 的极端压缩形式
- AI 的"幻觉" = 偏离可行区域的搜索结果
- 晶体化视角的独特贡献：传统 AI 工程方法论（MLOps、LLMOps）将"代码系统"和"模型系统"视为需要不同工具链管理的两类对象；GIM 将它们视为同一收敛谱系上的不同位置，统一在"压缩即理解"的框架下

## V. 工具支持

*GIM 概念如何映射到可构建的工具能力*

### 5.1 架构压缩工具（ArchGuard 当前能力）

- 源代码 → ArchJSON → 多层级图：实现"认知 MDL"的压缩
- 包/类/方法三级视图 = 不同粒度的压缩投影

### 5.2 架构度量工具（ArchGuard 扩展方向）

- 敏感度代理指标：git churn × 耦合度 → "Fisher 热力图"
- 本征维度代理：模块独立性评分
- 压缩质量评分：重复模式检测、循环依赖计数

### 5.3 AI 上下文生成器（ArchGuard × AI Agent）

- 为 AI Agent 生成机器可读的架构摘要
- 在 AI 修改代码前提供"可行区域边界"提示
- 变更风险评估：哪些修改可能偏离架构"测地线"

## VI. 限制与诚实声明

*方法论的边界*

### 6.1 GIM 不是什么

- 不是可量化验证的科学理论
- 不提供精确的数值阈值（"何时重构"没有公式）
- 不替代领域专家的判断

### 6.2 已知的认知偏差风险

- "锤子效应"：不是所有工程问题都适合几何隐喻
- 过度抽象的诱惑：GIM 本身也受"G 膨胀"约束
- 术语壁垒：几何/信息论词汇可能增加而非降低沟通成本

### 6.3 与 GIT 理论文献的关系

- GIT 文献（本文档目录下的历史版本）提供了数学灵感来源
- GIM 保留了 GIT 的核心直觉，但放弃了其"精确科学"的定位声明
- 未来如果出现可行的量化方法，GIM 可以回归理论框架
