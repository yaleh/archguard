## §0 本版本修订动机

v1.0 草案把"密集形式化记号（如类 Haskell）prompt 效果不错"这条经验总结性地扔进一个实验里。v1.1 通过以下三条修订升级：

1. **假设拆分**：这句经验纠缠了三个机制不同的假设（H-format / H-rewrite / H-info），对应三个独立实验。
2. **方法论锚定**：LLM 改写让混淆**转移进黑箱**，往返验证是唯一可靠的外部锚；推广到所有含 LLM 改写的臂。
3. **新增混淆分析**：密度/长度在信息固定下退化为一根轴（§2.2）；局部信息密度分布作为独立协变量显式引入（§3.3）；改写臂的跨家族偏差作为设计要求（§4.2）。

## §1 核心假设拆分（骨架）

"密集形式化记号帮模型"这句经验，纠缠了三个**机制不同、后续不同**的假设：

| 假设 | 主张 | 测哪个实验 |
|---|---|---|
| **H-format** | 同信息，*Haskell-ADT 编码本身*帮模型 | **Exp 1**（确定性渲染） |
| **H-rewrite** | *LLM 改写这个动作*（消歧/重构）帮模型，与目标记号无关 | **Exp 2**（改写 pass） |
| **H-info** | LLM 改写偷偷*改了信息*（补结构/删冗余），提升是信息变了 | **往返验证排除**（横贯） |

主观体验分不开这三个，LLM 替你改写让它们**更不可分**——消歧/重构/信息变动全发生在一个不透明的黑箱里。本方案的全部意义，是给这三个各配一个**外部锚**，把它们切开。

> **核心方法论原则**："逼作者想清楚"的混淆没有因为改写者从人换成 LLM 而消失——它**转移进改写模型、并变得更不透明**。唯一解药：Exp 1 用**确定性渲染 + 往返验证**从构造上消除黑箱；Exp 2 凡有 LLM 改写之处，改写输出**必须**跑往返验证，否则 H-info 无法与 H-rewrite 分离。

## §2 横贯方法论

### 2.1 规范结构 C 与往返验证（全实验通用）

**规范结构 $C$**（method 级类型化实体-关系图）是唯一信息真相。

**C 的 schema 须在执行顺序第 1 步冻结**（见 §8），先于任何渲染器编写。schema 须明确：

- 实体字段：`id`、`name`、`type`（class/interface/function/…）、`sourceFile`、`methods[]`（每个 method 含 `name`、`params[]`、`returnType`）
- 关系字段：`from`（entity id）、`to`（entity id）、`type`（枚举，预注册：`call`/`inheritance`/`composition`/`aggregation`/`dependency`/`implementation`）
- 可缺省字段的默认值（须显式声明，避免渲染器与解析器隐式分歧）

**确定性渲染** $r_f: C \to \text{format}_f$：纯函数，不增不删。  
**往返门控** $p_f(r_f(C)) = C$：无损 = 信息等价**被证明**，而非被断言。  
**往返 diff 的逻辑须预注册**：对 nullable 字段、顺序无关集合（relations 集合无序）、等价 id 别名，须在冻结文档中写明 diff 规则，否则往返检查会产生假阳性/阴性。

**[架构师修订 — diff 规则须在冻结文档中显式写到此粒度，以下为要求的最低规范：]**

预注册的 diff 算法须包含以下四条规则，并在 git tag 前以代码级伪码锁定：

1. **实体集合等价**：`C.entities` 对比用 `id` 为主键的集合等价（`Set<id>`），实体顺序无关；每个实体字段逐字段比对，类型枚举值须大小写规范化后再比对。
2. **关系集合等价**：`C.relations` 对比用 `(from, to, type)` 三元组构成的集合（`Set<(from,to,type)>`），顺序无关；若同一 `(from,to)` 有多条不同 `type` 的关系，均为独立成员。
3. **nullable 字段处理**：预注册每个可缺省字段的等价规则，例如 `params: null` ≡ `params: []`、`returnType: null` ≡ `returnType: "void"`；未在此规则中列出的 nullable 字段，`null` 与缺失视为**不等价**（保守规则，触发偏差记录）。
4. **id 别名规范化**：渲染器和解析器须对实体 id 使用同一规范化函数（如 `fullyQualifiedName.toLowerCase().trim()`）；规范化函数须在 schema 冻结步骤中以代码锁定，不得在后续修改。

**推广到 LLM 改写**（Exp 2）：LLM 改写的任何产物，也要解析回 $C'$ 并对照 $C$：

- $C' = C$ → 信息保住，提升可归 H-rewrite（改写动作本身）
- $C' \neq C$ → 信息变了，该实例归 H-info，**移出等价比较，单列报告**

### 2.2 密度与 token 数在信息固定下的关系

在 $C$ 固定的条件下，信息量是常数，信息密度 = 信息/token ∝ 1/token。这意味着：

- **密度与 token 数完全反相关**，二者在 Exp 1 的框架里不是两个独立变量，而是**同一根轴**。
- 分析策略：测量每种格式的 token 数；做"token 调整后排名"分析。若某格式 token 调整后仍赢 → **格式结构本身有效**，非省 token 的副产品；若调整后赢消失 → 其表观优势只是省 token。

**但总 token 数不足以控制全部混淆**：同 token 数下，不同格式的**局部信息密度分布**不同。例如，Haskell-ADT 把类型信息前置在构造子，重要关系集中在前段 token；JSON 把信息分散在 key-value 对里；Mermaid 把信息散在箭头标注里。注意力分配受局部密度分布影响，这是总 token 数无法控制的协变量。

因此，Exp 1 的协变量测量需包含**两层**：
1. 总 token 数（密度协变量）
2. **关键信息位置分布**：重要关系（call 边 / inheritance 边）在序列中的位置中位数，以及前 20% token 内出现的关键关系比例

## §3 Experiment 1：数据格式效应（隔离 H-format）

**唯一不用任何 LLM/人改写的实验**。全部格式由确定性转换器从 $C$ 渲染，信息逐位相同。

### 3.1 格式集（主集合：须无损往返）

按（熟悉度 × 嵌套/平 × 密度）铺格，每格占一个不同的角落：

| 格式 | 熟悉度 | 结构 | 局部信息分布 | 测什么 |
|---|---|---|---|---|
| JSON-邻接 | 高 | 嵌套 | 类型前置，嵌套深 | B4 赢家，基准 |
| JSON-边表 | 高 | 平 | 均匀分布 | vs 邻接 → 隔离解析负担（同熟悉度，仅嵌套异） |
| YAML | 高 | 缩进表义 | 缩进集中类型 | 是 JSON 特有还是结构格式普遍；缩进语义 |
| Markdown 表 | 高 | 平、表格 | 标题行前置，均匀 | 最熟 + 最平 + 共址 |
| Mermaid | 中 | 图-DSL 嵌套 | 箭头标注散布 | B4 输家；与 JSON-邻接正面对比 |
| **Haskell-ADT** | **中-低** | **嵌套、密集** | **构造子前置，类型集中** | 填"密集+嵌套+中低熟"空格；解耦密度与平坦 |
| 新造 DSL | 低 | 平 | 均匀 | 关键判别器：生但简 |
| NL-穷举 | 高（prose）| 平、啰嗦 | 均匀分布 | 无损逐边复述，测 prose 框架本身；同时作为 Exp 2 改写的基线起点（双重角色，见下方注释）|

**Haskell-ADT 的操作化（冻结）**：实体为 record/ADT 构造子，关系为类型化字段。例：

```haskell
data Renderer = Renderer
  { calls       :: [Pipeline]
  , implements  :: [IRenderer]
  , methods     :: [RenderDiagram, RenderSVG]
  }
```

天然无损往返（ADT 编码类型化图），是主集合合法成员，非有损参照。

**新造 DSL 的操作化（冻结）**：最小化语法，一行一条关系：

```
Renderer -calls-> Pipeline
Renderer -impl-> IRenderer
```

**砍掉的格式**：DOT（与 Mermaid 重复，"另一个中熟嵌套图-DSL"）、缩进树（与 YAML 重复，"另一个缩进层级"）。

**[架构师修订 — NL-穷举的双重角色须显式处理：]**

NL-穷举在本 proposal 中同时承担两个角色：(1) Exp 1 主集合格式之一（参与 8 格 Friedman 检验）；(2) Exp 2 所有改写臂的基线起点（"从 NL-穷举出发改写"）。这两个角色的数据是**同一批 prompt**，不会产生数据矛盾，但须在冻结文档中明确声明此双重角色，避免执行时混淆。具体要求：
- Exp 1 中 NL-穷举的答题数据（68 任务 × 模型 × k=5）同时充当 Exp 2 "基线臂"的数据，**不重复跑**。
- 若 Exp 1 和 Exp 2 使用不同的 prompt 结构（如 Exp 2 有额外的指令前缀），则两者的 NL-穷举 prompt **不等价**，须分开跑并在文档中标注差异。
- 此决策须在冻结前明确，写入 §6 配置表。

### 3.2 非等价参照轨（往返有损，单独报告）

- **NL-摘要**（有损流畅抽象）：单独轨；问"流畅摘要在 C 类综合任务上换不换得过无损结构"。本质有损，**不进入主集合统计，单独报告**。
- **原始源码**：信息超集 + 需推断，仅作"不预抽取"参照点，不进主集合。

### 3.3 关键对比（预先设计，冻结）

| 对比 | 什么变了 | 什么不变 | 推断什么 |
|---|---|---|---|
| JSON-边表 vs JSON-邻接 | 嵌套/平坦 | 熟悉度（同为 JSON）| 解析负担 |
| 新造 DSL vs JSON-边表 | 熟悉度 | 都是平坦简洁 | 熟悉度 vs 简洁度（若新造 DSL 更简却输 JSON → 是熟悉度，非简洁度）|
| Haskell-ADT vs JSON-边表 | 密度+嵌套、熟悉度降 | 信息量相同 | token 调整后看谁赢 → 密度本身是否独立有效 |
| Haskell-ADT vs Mermaid | 熟悉度、密度 | 都是嵌套型 | 熟悉度的净效应 |
| 任意格式 × A/B/C 任务类 | 任务类型 | 格式 | 格式 × 任务类交互 |

**[架构师修订 — "新造 DSL 更简"须预注册操作化定义：]**

"新造 DSL 更简"是 H-pretrain 的核心判定前提，但"简"目前未量化。须在冻结文档中预注册以下操作化定义：

- **词汇表大小（vocabulary size）**：格式语法中的关键字 + 分隔符种类数。新造 DSL 预期 ≤ 5 个非字母符号（`->`、`:`），JSON 有 `{}`、`[]`、`:`、`"`、`,` 共 5 种（相近）；Mermaid 有 `graph`、`subgraph`、`-->`、`-.->` 等 ≥ 10 种关键字。
- **每条关系的平均 token 数**：预计算每格式表达一条关系所需的 token 数（mean ± std across 所有关系）；新造 DSL 目标：≤ 5 tokens/关系。
- **判定规则（冻结）**：若新造 DSL 的词汇表大小 ≤ JSON-边表 且每关系 token 数 ≤ JSON-边表 × 0.8，则判定"新造 DSL 更简"成立；否则 H-pretrain 的简洁度前提不成立，该对比须重新标注（不是"简但输 JSON"，而是"简洁度相当"）。

此定义须在第 2 步锁定，不得在看到结果后修改。

### 3.4 被试、粒度、任务

- **被试**：ArchGuard method 级 $C$ + 1–2 私有库（泛化，须 TypeScript 项目，callgraph 工具覆盖；若非 TypeScript，须在冻结前预注册等价 callgraph 工具）。
- **粒度**：钉死 method 级（与 v2.2 的粒度结论正交；可选 package 级交互臂，但主分析在 method 级）。
- **任务**：复用 v2.2 的 68 题（A25 / B22 / C21）+ **解析检查子任务**（让模型把格式解析回 $C$ 的子集，用于分离"误读格式"与"推理错"）。评分：set → median-F1；categorical → majority-EM；numeric → median-EM，k=5。

**[架构师修订 — 任务复用泄漏风险（须在冻结前决策）：]**

复用 v2.2 的 68 题存在两类泄漏风险，均须在冻结前明确处置：

**风险 1 — 同模型跨实验记忆（上下文层面）**：无状态 API 调用不存在跨 session 记忆，此风险**可排除**（每次调用均为独立 prompt）。

**风险 2 — 任务内容进入模型训练数据（RLHF/微调层面）**：若 v2.2 的实验 prompt 通过 API 使用数据反馈进入了模型的 RLHF 训练，复用同一题目集的结果会产生系统性乐观偏差（模型已在这些题型上被强化）。这对 Claude 系列模型风险较高（Anthropic 使用 Constitutional AI + RLHF，API 数据可能进入训练）。**此风险在当前实验规模下无法从构造上排除**，须在报告中明确声明为已知局限。

**处置选项（须在冻结时选定其一）**：
- (A) **接受复用 + 声明局限**：在报告中标注"任务集与 v2.2 完全重叠，结论适用于该任务集，跨任务集泛化须独立验证"。优点：直接可比 v2.2；缺点：无法排除 RLHF 偏差。
- (B) **混入新题（建议 ≥20 道）**：在 68 题基础上新增与 v2.2 不重叠的题目，报告分层结果（复用题 vs 新题）；若两层结果一致，则 RLHF 偏差假说削弱。
- (C) **换用在 v2.2 实验期间未通过 API 调用的模型族**：如使用仅本地部署的开源模型（Llama 系列）作为辅助答题模型，其训练数据不含 API 反馈。

### 3.5 混淆控制

| 混淆源 | 控制方式 |
|---|---|
| 信息不等价 | 往返门控（§2.1），未过者排除 |
| 熟悉度 | 有 logprob 的模型上测每格式的**困惑度**作协变量（注：困惑度是模型专属，须逐模型测量；见下方 API 可用性注释）|
| 解析负担 | 嵌套深度 + 每边 token 数 + 分隔符密度 + 缩进是否语义化（4 个可测指标）|
| 总 token 数 | 测量 + token 调整后分析 |
| 局部信息密度分布 | 见 §2.2，关键关系位置中位数 + 前 20% 比例 |
| 顺序效应 | 任务顺序随机化 |
| 模型单点 | 两个已确认答题模型：claude-haiku-4-5-20251001（Claude 家族）+ glm-4.5-flash（Zhipu 家族）；跨家族一致方可推广 |
| 思考型 token 耗尽 | max_tokens=8192（v2.2 D-71.2 教训）|

**[架构师修订 — 困惑度协变量可操作性缺口：]**

"困惑度"在此用作熟悉度的代理，但其可操作性取决于 logprob API 的可用性：

- **Claude（Anthropic API）**：**不提供 logprob**，无法计算 perplexity。若答题模型之一为 Claude，困惑度协变量对该模型**不可测量**，H-归因回归中的"困惑度系数"对 Claude 的答题数据无法填入。
- **OpenAI GPT 系列**：提供 `logprobs` 参数，可算 per-token log-prob，进而得 perplexity。
- **Deepseek / Qwen**：须在冻结前确认 API 是否暴露 logprob；v2.2 使用了 deepseek-v4-flash，但该 API 是否返回 logprob 未经确认。

**强制要求**：在执行顺序第 2 步锁定模型列表时，须同时确认每个模型的 logprob 可用性，并在冻结文档中记录。无 logprob 的模型，"熟悉度"列改用**替代代理**（见 §10 开放问题 Q2），且回归方程中该协变量须标注为 NA 并在报告中说明。不得在实验后才发现 logprob 不可用。

### 3.6 事前预测（冻结）

- **H1**：格式主效应显著（Friedman p < 0.05/BH-FDR）
- **H-parse**：平坦格式 > 嵌套格式（同熟悉度条件下），即 JSON-边表 > JSON-邻接
- **H-pretrain**：熟悉格式 > 生造格式（JSON-边表 > 新造 DSL），且新造 DSL 更简洁却仍输 JSON → 熟悉度是净效应而非简洁度
- **H-dense**：Haskell-ADT token 调整后仍赢 JSON-边表 → 密集形式化编码本身有效（独立于省 token）
- **H-interact**：格式 × 任务类交互显著（A 类 vs B/C 类对格式的敏感度不同）
- **H-归因回归**：混合效应回归 `accuracy ~ 困惑度 + 解析负担 + tokens + 局部密度分布 + 任务类`，困惑度系数与解析负担系数均显著

### 3.7 统计

- Friedman + 成对 Wilcoxon（pairwise）+ BH-FDR（q=0.05），跨所有 8 个主集合格式
- 混合效应回归（任务随机效应，格式 × 任务类固定效应）
- Token 调整后排名分析（控制 token 数后，格式排名是否存活）
- 逐模型报告（跨模型稳定的结论才可做行动性推断）

**[架构师修订 — 样本量估算与统计功效分析（缺失，须在冻结前补全）：]**

当前 proposal 完全缺少功效分析。以下为必须在冻结前完成的估算：

**Exp 1 调用量估算**：
- 主集合：8 格式 × 68 任务 × 2 答题模型 × k=5 = **5440 次 LLM 调用**（最低配置，仅 2 模型）
- 若 3 模型：8160 次；若加解析检查子任务（每任务 1 次）：+1088 次
- 参照轨（NL-摘要 + 原始源码）：2 × 68 × 2 × k=5 = 1360 次额外调用
- **Exp 1 合计最低估：~6800 次**（2 模型，含参照轨）

**Exp 2 调用量估算**：
- 改写步：3 改写臂 × 68 任务 × 2 改写模型 × k=5 改写 = 2040 次（2×2 设计下）
- 答题步：5 臂 × 68 任务 × 2 答题模型 × k=5 = 3400 次
- **Exp 2 合计：~5440 次**（2×2 设计最低配置）

**总调用量：≥13,328 次**（最低配置，2 模型，不含 Exp 3；详见 §10 Q1 明细表）。须在冻结前预估 API 费用上限并获得批准。

**最小可检测效应（MDE）**：
- v2.2 B4 观测到 Δ=38.4%（L4 vs L3），Wilcoxon p=7.6×10⁻⁸，n=68 任务，k=5
- 对于 8 格式的 Friedman 检验（28 成对比较，BH-FDR q=0.05），以 80% 功效，使用 v2.2 同样的任务集（n=68），可检测的最小 Δ 估计在 **15–20%** 范围（需用 bootstrap 精确估算）
- **若目标是检测 Δ≈10% 的格式差异，n=68 可能不足**——须在冻结前做 bootstrap 功效模拟（使用 v2.2 的分数分布作为先验），并预注册最小可接受功效阈值（建议 80%）
- 若功效不足，须在冻结前决定：(a) 扩大任务集；(b) 提高 k；(c) 接受更低功效并在报告中标注；不得在看到结果后才补充此声明

### 3.8 Exp 1 决策表

| 条件 | 判定 |
|---|---|
| 某格式往返不无损 | 排除出主集，记录偏差 |
| H1 成立 | 格式重要，量化 Δ，给 ArchGuard **默认格式建议** |
| H-parse 显著 / H-pretrain 不显著 | 机制 = 解析负担 → 工程行动：压平语法 |
| H-pretrain 显著 / H-parse 不显著 | 机制 = 熟悉度 → 工程行动：对齐预训练分布，避免自造 DSL |
| 两个系数均显著 | 两者皆有，报相对权重 |
| **H-dense 成立（token 调整后 Haskell-ADT 仍赢）** | **密集形式化编码本身有效**，非仅省 token |
| **H-dense 不成立（调整后赢消失）** | 表观优势 = 省 token，记号本身无魔法 |
| H-interact 显著 | 格式路由须任务类感知（叠加 v2.2 的按类粒度路由）|
| H1 不成立 | 格式效应不显著；在 ArchGuard 当前规模上，格式选择不影响准确率 |

## §4 Experiment 2：LLM-改写 pass（隔离 H-rewrite 与 H-info）

测你**实际做过**的那件事：LLM 把 prompt 改写成 Haskell。问：提升是**记号**（H-format）、**改写动作本身**（H-rewrite），还是**信息被改了**（H-info）？

### 4.1 臂设计

从同一基线表示（prose 版的 $C$，即 NL-穷举）出发：

| 臂 | 生产方式 | 目标格式 | 往返要求 |
|---|---|---|---|
| **确定性-Haskell** | 渲染器 $r_\text{haskell}$（Exp 1 的同一渲染器） | Haskell-ADT | 无损（已在 Exp 1 验证）|
| **改写-Haskell** | 改写模型（见 §4.2） | Haskell-ADT | **强制往返**，$C' \neq C$ 的实例单列 |
| **改写-JSON** | 改写模型 | JSON-邻接 | **强制往返** |
| **改写-清理 prose** | 改写模型 | 干净大白话（无结构改变）| **强制往返**（检查是否增删关系）|
| **基线** | 无改写 | NL-穷举（原 prose） | — |

### 4.2 改写模型的跨家族要求（**设计强制项**）

若改写模型与答题模型来自同一家族，二者共享预训练分布偏好，改写输出会天然与答题模型的内部表示对齐——这既不是 H-format（记号功劳），也不是 H-rewrite（重排功劳），而是**同家族表示对齐**，往返验证无法排除它。

**设计要求**：

- 改写模型与答题模型**使用不同家族**（e.g. 用 Gemini 或 Qwen 改写，用 Claude 答题）
- 设计 2×2 交叉：{改写家族 A，改写家族 B} × {答题模型 X，答题模型 Y}
- 若 2×2 不可行，至少两组（改写家族 A + 答题 X；改写家族 B + 答题 X），看家族交互效应
- 改写模型须在冻结文档中预注册，不得事后更换

**[架构师修订 — 2×2 交叉设计的统计检验逻辑（缺失，必须补全）：]**

2×2 设计要求在预注册中写明以下内容，否则"改写家族交互显著"这一判定条件（§4.4）无法执行：

1. **交互效应检验方法**：对 2×2 设计（改写家族 × 答题模型），用**双因素混合效应 ANOVA 的交互项**（`rewriter_family × answer_model`），或等价的 Wilcoxon × Stouffer 合并检验；检验统计量须在冻结文档中指定，不得事后选择。
2. **主效应与交互效应的判定顺序**：先检验交互效应（α=0.05，BH-FDR 保护），若显著则报告各格结果（不可合并）；若不显著则合并跨改写家族的结果，报告答题模型主效应。
3. **2×2 所需样本量**：68 任务 × k=5 = 340 观测/格，共 4 格 = 1360 观测；以 v2.2 的方差水平（σ ≈ 0.25）估计，检测 10% 交互效应需要的任务数约为 80–100。68 题在 2×2 设计中检测交互效应的功效**有限**，须在冻结前做功效模拟并接受或调整设计。
4. **降级条件（预注册）**：若 2×2 完整设计不可行（某格的改写模型无 API 访问权限），降级到"1×2 + 家族固定为 A"设计，但须在**冻结时**即声明降级，不得在执行中途切换。

**改写 prompt**：预注册并冻结；k=5 采样多份改写（估改写方差）；改写 prompt 应只指定目标格式，不允许模型"补充""推断""删除模糊信息"（此类指令会触发 H-info）。

### 4.3 读法（预注册的推断逻辑）

**对比 1**：改写-Haskell vs 确定性-Haskell（同记号，仅生产方式异）

- 往返通过（$C'=C$）且改写-Haskell > 确定性-Haskell → **H-rewrite**（LLM 改写在信息不变的前提下带来额外收益，如命名优化/排序改善）
- 往返通过且无显著差 → 改写的重排效应为零，格式本身是唯一变量，归 H-format
- 往返不通过（$C' \neq C$）且改写-Haskell > 确定性-Haskell → **H-info**（提升来自信息变化）

**对比 2**：三个改写臂互比（改写-Haskell / 改写-JSON / 改写-清理 prose）

- 三者改进相近（无显著差）→ **H-rewrite**：是**改写 pass 本身**在起作用，目标记号无关。**可行动结论：给 prompt 上一道 LLM 改写/批评 pass，无需指定 Haskell。**
- 改写-Haskell 在改写臂里特别赢 → 记号有特异性，再回 Exp 1 的确定性-Haskell 交叉确认（排除重构功劳）
- **改写-清理 prose 与其他改写臂无差** → 从来不是 Haskell，是改写动作

**关键判别器**：改写-清理 prose 臂。若改成大白话也改进同样多，则记号与格式 incidental，改写 pass 本身是杠杆点。

### 4.4 Exp 2 决策表

| 条件 | 判定 |
|---|---|
| 改写-Haskell > 确定性-Haskell（往返通过）| H-rewrite 成立；改写有净效应 |
| 三个改写臂无显著差 | H-rewrite + 记号无关；改写 pass 为行动点 |
| 改写-清理 prose = 改写-Haskell | 提升来自改写动作，不来自 Haskell 记号 |
| 改写-Haskell > 改写-清理 prose | 记号有特异性；回 Exp 1 确定性臂交叉确认 |
| 任意改写臂往返不通过 | 该实例归 H-info，单列；主比较仅用往返通过的实例 |
| 改写家族交互显著 | 存在家族偏差；报告交叉设计各格中的数字，稳定结论须跨家族一致 |

## §5 Experiment 3：指令编码镜像（暂缓，待 Exp 1/2 后预注册）

Exp 3 是 Exp 1 的对偶：固定数据格式（用 Exp 1 赢家），变**任务/指令的编码**——大白话 vs 类型签名规约（`task :: Context → Constraints → Output`）vs 函数管线（`parse >>> validate >>> render`）。

**为何暂缓**：

- Exp 1 的结论会直接影响 Exp 3 的设计。若 Exp 1 发现"熟悉度主导、记号无关"，则 Exp 3 里类型签名 vs 大白话的判定框架需要调整；若 Exp 2 发现"改写 pass 本身有效"，则 Exp 3 需要新增"把指令改写成类型签名"vs"把指令改写成大白话"的对照臂。
- Exp 3 的指令等价验证无法做到 Exp 1/2 级别的往返无损（指令描述程序，非数据结构；语义等价需用 LLM 确认，结论强度低）。
- 在 Exp 1/2 预注册冻结后并行跑 Exp 3，意味着 Exp 3 无法吸收前两个实验的发现。

**暂缓策略**：Exp 1/2 执行并出具报告后，基于其结论重新设计 Exp 3，再单独预注册冻结。本 proposal 仅记录 Exp 3 的意图和已知限制，不包含其可冻结的设计。

**Exp 3 的已知诚实限制**（供后续版本继承）：

- 指令不是图，无干净往返；控法只能用：从固定指令语义规约编译各风格 + 回译 + 多评审近似确认
- 此验证比数据往返弱，须在报告中显式标注结论强度低于 Exp 1/2
- 指令编码 × 数据格式的交互效应只能在 Exp 1/2 完成后才能正确设计

## §9 产出与与先序实验的关系

**本实验的直接产出**：

- Exp 1 → **ArchGuard 默认数据格式建议**（哪个格式最优 + 机制是熟悉度还是解析负担）

- Exp 2 → **H-info 量化**（改写导致的信息变化量与提升量的相关性）

**与 granularity-v2.2 的叠加**：v2.2 确定了"按任务类路由粒度"（**A→L0（文件列表），B/C→L3（method 级）**）；本实验叠加"在最优粒度内用最优格式"。两者组合构成完整的桥-设计规则：先按任务类选粒度，再按本实验结论选格式。

注意：A 类最优为 L0（无内容的文件列表）意味着 A 类任务在本实验（固定 method 级）下**不是**最重要的测试对象——method 级格式对 A 类影响可能有限。本实验若按 v2.2 结论应重点关注 B/C 类任务在 method 级的格式敏感性；A 类数据可作辅助分析，但不应驱动主结论。

**本实验明确不碰**：维度/几何（v2.2 已否定）；粒度选择（钉死 method 级，粒度问题归 v2.2）。

**Q2 — 困惑度协变量的替代方案是否可接受？（须决策）**

**2026-06-12 探针更新**：两个确认答题模型的 logprob 状态：
- **claude-haiku-4-5-20251001**：Anthropic API 不提供 logprob，**无法测 perplexity**。
- **glm-4.5-flash**：Zhipu API 未暴露 logprob（LiteLLM 代理层未见此字段），**按不可用处理**，冻结前须最终确认。

两个主答题模型均无 logprob，方案 B 或 C 为实际可选项。候选替代方案：

- (A) **用次级模型（如 GPT-4o）在各格式上测困惑度，作为格式熟悉度的代理**：可操作；但 GPT-4o 困惑度不代表 haiku/GLM 熟悉度，结论可迁移性弱。
- (B) **完全放弃困惑度协变量，改用纯结构指标（嵌套深度、分隔符密度等）**：H-归因回归改为 `accuracy ~ 解析负担 + tokens + 局部密度分布 + 任务类`；"困惑度系数"预测降级为探索性。**鉴于两个主模型均无 logprob，此方案操作成本最低。**
- (C) **加入 GPT-4o 作为第 3 个答题模型**（有 logprob）：对 GPT-4o 可做完整归因回归；对 haiku/GLM 做无困惑度简化回归。增加调用成本约 5440 次（Q1 已计入最低估算外）。

须选定一个方案并写入冻结文档，不得在分析阶段才决定。

**Q3 — 任务泄漏处置方案选择（须决策：A / B / C）**

见 §3.4 修订。三个选项：
- (A) 接受复用 + 声明局限（最简单，但可能有 RLHF 偏差）
- (B) 混入 ≥20 道新题（推荐，成本中等，可分层验证）
- (C) 加入本地开源模型作为辅助答题模型（技术成本高）

**架构师建议选 (B)**：新增 20 道覆盖 B/C 类任务的新题（因 v2.2 结论表明 method 级对 B/C 类最关键），新旧题分层报告。若两层结果方向一致，泄漏假说被实质削弱。

**Q4 — Haskell-ADT 往返验证的"天然无损"声明是否有实证支撑？（须验证）**

§3.1 声称 Haskell-ADT"天然无损往返（ADT 编码类型化图）"，但这是设计意图，不是实证结论。`methods` 字段编码为 `[RenderDiagram, RenderSVG]`（类型名列表），解析器能否从类型名还原回 method 的 `params[]` 和 `returnType`？若 method 详情被压缩为单个类型标识符，则往返**有损**，Haskell-ADT 应归入有损参照轨而非主集合。

须在步骤 3（写渲染器 + 解析器）完成后、步骤 4（往返门控）执行前，对 Haskell-ADT 的往返无损性进行 smoke test，并在 smoke test 通过后才可将其保留在主集合。若往返有损，须在冻结前重新设计 Haskell-ADT 格式（确保 method 详情可还原），或将其移至有损参照轨并调整 §3.3 的关键对比设计。

**Q5 — "改写-清理 prose"的往返验证是否可行？（须确认）**

§4.1 要求"改写-清理 prose"也做往返验证（检查是否增删关系）。但"干净大白话"不是结构化格式，**没有确定性解析器**——无法像 JSON 或 Haskell-ADT 那样写 $p_f$。往返验证此臂的唯一方法是：用 LLM 把大白话解析回 $C'$（即 LLM-as-parser），但这引入了另一个不透明黑箱。

须在冻结前决定：
- 是否接受"LLM-as-parser 解析大白话→C'"？若接受，须预注册解析 prompt，并说明此往返的结论强度低于结构格式的确定性往返。
- 或将"改写-清理 prose"的往返要求降级为"人工抽样审核 10% 实例"，明确标注此臂的 H-info 控制强度低于结构改写臂。
