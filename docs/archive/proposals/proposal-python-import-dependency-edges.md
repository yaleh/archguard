# Proposal: Emit Inter-Package Dependency Edges for Python

**Status**: Draft
**Date**: 2026-03-13
**Author**: ArchGuard Team

---

## 1. Background

### Current state

Python package overview diagrams show 0 inter-package relations. Confirmed on lmdeploy: 96 package
entities, 0 relations in the rendered package diagram.

The mapper (`src/plugins/python/archjson-mapper.ts`) does call `createImportDependency()` for every
`PythonRawImport` on a module. However, the emitted `target` is the raw `imp.module` string
extracted verbatim from the AST — values such as `lmdeploy.models.base`, `.utils`, or
`..common.helper`. These strings are never registered as entity IDs in the ArchJSON. The diagram
generator therefore finds no matching node for any `dependency` relation target, and the package
graph is rendered with isolated nodes.

Four specific failure modes combine to produce the 0-relation result:

1. **Entity ID mismatch**: Entities have IDs of the form `lmdeploy.models.base.MyClass` (dotted
   module path + class name, generated by `generateEntityId`). Package-level pseudo-entities are
   created for the relation source (via `generateModuleId`, e.g. `lmdeploy.models.base`), but the
   target of each relation is also a raw module path — which may or may not correspond to a
   `generateModuleId` value for any parsed module. The package diagram renderer needs source and
   target to be package IDs, not entity IDs, so even when the strings happen to match, the
   renderer cannot use entity-level relations for package-level arrows.

2. **Relative imports stored verbatim**: For `from . import X`, the tree-sitter `module_name`
   field is always `"."`. The bridge stores `imp.module = "."`. `createImportDependency` then emits
   a relation with `target: "."`, which is meaningless without relative-import resolution logic.
   Similarly, `from .utils import X` stores `".utils"` and `from ..common.helper import Y` stores
   `"..common.helper"` — all requiring resolution against the source file's location before they
   can be matched to a known module ID.

3. **No intra-package deduplication or stdlib filtering**: Every import from every module produces
   a relation entry, including imports of `os`, `sys`, `typing`, and third-party packages (e.g.
   `torch`, `numpy`). These targets are not in the ArchJSON entity set; they would appear as
   dangling nodes if a renderer attempted to include them.

4. **Aliased imports produce unparseable module strings**: For `import X as Y`, the bridge stores
   `imp.module = "X as Y"` (the full `aliased_import` node text verbatim). The ` as Y` suffix
   is never a valid module path, so the string never matches any entry in the module index. All
   aliased bare-form imports (`import pkg.module as alias`) are silently lost.

### Data already available in memory

All raw data needed to produce correct package-to-package edges already exists at mapping time:

- `PythonRawModule.imports: PythonRawImport[]` (`src/plugins/python/types.ts:124`): every parsed
  module carries its full import list, including absolute (`lmdeploy.models.base`) and relative
  (`.utils`, `..common.helper`) forms, as extracted by `TreeSitterBridge.extractImport()`.
- `PythonRawModule.filePath: string` (`types.ts:121`): the absolute path to the source file.
- `PythonRawModule.name: string` (`types.ts:120`): the short filename stem (e.g. `base`), set by
  `extractModuleName()` in the bridge.
- `workspaceRoot: string` passed into `mapModules()`: the project root used to derive stable dotted
  module IDs via `generateModuleId()`.
- The full `modules: PythonRawModule[]` list is passed at once to `mapModules()`, so a lookup from
  dotted module path → known module is cheaply constructable in a single pass.

The `PythonRawImport` interface (`types.ts:110-114`) holds:

```typescript
interface PythonRawImport {
  module: string;       // e.g. "lmdeploy.models.base", ".", ".utils", "..common"
  alias?: string;
  items?: Array<{ name: string; alias?: string }>;
}
```

### Why this matters

Without inter-package dependency edges:

- The Python package diagram is a set of disconnected nodes providing zero architectural insight.
- ArchGuard's primary value proposition — visualising how modules depend on each other — is absent
  for all Python projects.
- MCP query tools (`getDependencies`, `getDependents`, cycle detection) return empty results for
  every Python entity because the ArchJSON relation graph has no usable edges.

## 2. Available Data

### What the Python plugin already collects

During `TreeSitterBridge.parseCode()`, every top-level `import_statement` and
`import_from_statement` node is extracted and stored in `PythonRawModule.imports`. For lmdeploy
with 96 packages, each module typically has 5–20 import statements, giving thousands of raw import
records available before `mapModules()` is called.

The tree-sitter node captures (verified against tree-sitter-python grammar via live AST inspection):

| Python source | `imp.module` value | Notes |
|---|---|---|
| `import lmdeploy.models.base` | `lmdeploy.models.base` | |
| `import lmdeploy.models.base as base` | `lmdeploy.models.base as base` | Bridge captures the full `aliased_import` node text verbatim; the ` as base` suffix must be stripped before lookup — **not currently handled, requires fix in resolution logic** |
| `import os, sys` | `os` | Only the first `name` child is captured; `sys` is silently dropped — minor limitation, acceptable for package-level graphs |
| `from lmdeploy.models.base import MyClass` | `lmdeploy.models.base` | |
| `from . import utils` | `"."` | `module_name` field is always `"."` for bare relative imports; the `""` value does **not** occur |
| `from .utils import X` | `.utils` | |
| `from ..common.helper import Y` | `..common.helper` | |
| `import os` | `os` | |
| `from typing import Optional` | `typing` | |

`generateModuleId()` (`archjson-mapper.ts:308-316`) already produces the correct dotted module path
for any parsed module given its `filePath` and `workspaceRoot`. The lookup keys that the new
resolution logic must match against are exactly what `generateModuleId` produces.

### What `ArchJsonMapper` currently does with imports

`mapModule()` calls `createImportDependency()` for every import, which calls
`createExplicitRelation('dependency', sourceId, imp.module, ...)`. The target is the raw
`imp.module` string — never resolved to a known module ID. No filtering is applied. The current
implementation is a structural stub that is semantically incorrect.

## 3. Goals

After this change:

- For each `PythonRawModule` M (source file F), and for each `PythonRawImport` I in `M.imports`:
  - Resolve `I.module` to a dotted module path T (handling relative imports using F's position
    within the package tree).
  - If T matches the `generateModuleId()` of another module in the same project, emit:
    ```
    Relation { id: "srcModId_dependency_tgtModId", type: 'dependency',
               source: generateModuleId(M), target: generateModuleId(target_module),
               inferenceSource: 'explicit' }
    ```
  - Stdlib imports, third-party imports, and imports with no matching project module are silently
    skipped.
  - Self-import edges (source === target) are guarded.
  - Duplicate edges are deduplicated.

- lmdeploy result expectation: ~200–400 unique inter-package `dependency` relations (based on the
  density of inter-module imports in a typical LLM inference framework).

- Existing `inheritance` relations are unaffected.

## 4. Solution Design

### Option A: Resolve imports inside `mapModules()` using already-parsed module list (recommended)

**Mechanism**: `mapModules()` already receives the full `modules: PythonRawModule[]` array. Before
the per-module mapping loop, build a lookup map from dotted module path → `generateModuleId`
value, covering all parsed modules. Then, for each module's import list, resolve each import to a
known module ID and emit a `dependency` relation if resolved.

**Relative import resolution**: Given a source file at `lmdeploy/models/base.py` and a relative
import `from .utils import X`, the relative module is `.utils`. Resolution: strip the leading dots
(one dot = same package directory), get the directory path of the source file
(`lmdeploy/models`), and append the remainder (`utils`) to get `lmdeploy.models.utils`. This
dotted path is then looked up in the module map.

For `from .. import common` (two dots = parent package directory): strip two dots, move up one
directory level (`lmdeploy`), and check if `lmdeploy.common` is a known module.

**`__init__.py` merging**: `lmdeploy/models/__init__.py` maps to the module ID `lmdeploy.models`
(via `generateModuleId`, which strips `/__init__` from the path). An import of `lmdeploy.models`
from elsewhere should resolve to this module ID. The existing `generateModuleId` logic already
handles this correctly via the `normalised.replace(/(\/|^)__init__$/, '')` line.

**What changes**:
- `ArchJsonMapper.mapModules()`: before the per-module entity loop, build `modulePathIndex:
  Map<string, string>` mapping `dottedModulePath → moduleId` for all modules. Pass this map to an
  enhanced `createImportDependency()`.
- `ArchJsonMapper.createImportDependency()`: replace the current stub with resolution logic that
  (a) strips stdlib/third-party candidates by checking map presence, (b) resolves relative imports
  using the source file path and the module path index, (c) emits a relation only when the target
  is found in the index.
- Add a private `resolveRelativeImport(relativeModule, sourceFilePath, workspaceRoot)` helper.
- Add a private `isKnownInternalModule(dottedPath, modulePathIndex)` guard (single map lookup).

**Pros**:
- Zero new parsing passes. All data is already in memory.
- Relative import resolution is pure string arithmetic on paths already available.
- Stdlib/third-party filtering is implicit: if a module is not in the project, it is not in the
  index, and no relation is emitted.
- No new fields on any raw type; no changes to the tree-sitter bridge.
- Fully synchronous — `mapModules()` stays synchronous.
- Change is self-contained in `archjson-mapper.ts` (~65–90 new lines).

**Cons**:
- `mapModules()` makes one full pass over `modules` to build the index before the mapping loop.
  This is O(N) and negligible for any real project.
- Relative import resolution requires knowing the source file's directory relative to
  `workspaceRoot`. When `workspaceRoot` is `undefined`, relative imports cannot be resolved and
  are silently skipped (acceptable — the single-file `parseCode()` path has no package context).

---

### Option B: Re-parse source files for import statements in the mapper (post-hoc)

**Mechanism**: `mapModules()` emits entities as today, then a second pass re-reads each source
file from disk to extract import lines with a regex, outside of tree-sitter.

**Pros**:
- Could be implemented without modifying the bridge or raw types.
- Could potentially catch import statements that tree-sitter misses (e.g. inside `if TYPE_CHECKING`
  blocks, though those are import-time-only).

**Cons**:
- Re-reading files from disk is strictly worse than using `PythonRawModule.imports`, which is
  already populated.
- `mapModules()` would need to become `async` (or use synchronous `fs.readFileSync`), which breaks
  the current call chain.
- The mapper (`archjson-mapper.ts`) would need to import `fs-extra`, violating its design as a
  pure data-transformation class.
- Duplicates the import extraction already done by the tree-sitter bridge.
- Any file read failure would silently drop edges or propagate exceptions.

**Verdict**: Rejected. Strictly dominated by Option A on every axis.

---

### Option C: Use tree-sitter bridge to emit import edges during initial parse

**Mechanism**: Extend `TreeSitterBridge.parseCode()` to return structured import records that
already classify imports as `internal`, `relative`, `stdlib`, or `external`, and resolve relative
imports to absolute dotted paths during parse time.

**Pros**:
- Import classification happens once, at parse time, with full AST context.
- Would enable future features (e.g. `TYPE_CHECKING` block detection, `__future__` annotations)
  more cleanly.

**Cons**:
- Requires passing `workspaceRoot` and a module index into the bridge — but the bridge is a
  single-file parser with no awareness of other modules. The module index does not exist until
  all files are parsed.
- This is a chicken-and-egg problem: resolving relative imports requires knowing the full module
  list, which only exists after all files are parsed. The bridge runs per-file before the full
  list is assembled.
- Adds cross-module state to what is currently a stateless per-file parser.
- Extends `PythonRawImport` with new classification fields, widening the interface unnecessarily
  for a concern that can be handled in `mapModules()` with existing data.

**Verdict**: Rejected. The bridge should remain a stateless per-file parser. Cross-module
resolution belongs in the mapper, which has global scope.

---

## 5. Recommended Option

**Option A** is the correct implementation.

The fundamental insight is that this is not a data-collection problem — it is a data-wiring
problem. The import strings are already in `PythonRawModule.imports`. The module identities are
derivable from `PythonRawModule.filePath` + `workspaceRoot` via the existing `generateModuleId()`
function. The full module list is already passed to `mapModules()` in one call. The only missing
piece is:

1. A pre-pass to index all known module IDs.
2. Resolution logic inside `createImportDependency()` that uses the index instead of the raw
   import string.

The resulting code addition is ~65–90 lines entirely within `archjson-mapper.ts`, with no new
dependencies and no interface changes outside that file.

## 6. Python-Specific Challenges

### 6.1 Relative imports (`from . import`, `from .. import`)

Python's relative import syntax is unique to the language. `from . import utils` means "import
`utils` from the current package directory". `from .. import common` means "import `common` from
the parent package directory".

The tree-sitter bridge records the `module_name` node content verbatim. For `from .utils import X`,
this is `.utils`. For `from . import X` (bare relative), the `module_name` field in tree-sitter's
Python grammar is always `"."` — the empty-string case does not occur in practice.

**Resolution algorithm** (for a source file at absolute path `sourceFilePath`):

1. Count leading dots: `dotCount = imp.module.match(/^\.+/)?.[0].length ?? 0`.
2. If `dotCount === 0`, the import is absolute — proceed to step 5.
3. Compute the base directory: start at `path.dirname(sourceFilePath)`, then move up
   `dotCount - 1` levels (one dot = same package, so no `dirname` elevation; two dots = parent
   package, so one `dirname` elevation).
4. If the import has a suffix after the dots (e.g. `.utils` → suffix `utils`, `..common.helper`
   → suffix `common.helper`), append the suffix path to the base directory.
5. Compute the dotted module path using `generateModuleId`-style logic:
   `path.relative(workspaceRoot, resolvedDir).replace(/\//g, '.')`.
6. Look up in the module path index.

When `workspaceRoot` is undefined (single-file API), relative imports are not resolvable and are
skipped.

### 6.2 `__init__.py` package merging

`lmdeploy/models/__init__.py` produces a module with `generateModuleId()` returning `lmdeploy.models`
(the trailing `/__init__` is stripped). Another module importing `from lmdeploy.models import X`
or `from lmdeploy import models` should resolve to the same `lmdeploy.models` ID.

The module path index built in `mapModules()` must key both `lmdeploy.models` (the `__init__.py`
entry) and handle the case where an import of `lmdeploy.models.SomeClass` should match the
`lmdeploy.models` module when no `lmdeploy/models/SomeClass.py` exists.

The safe resolution rule: after looking up the full dotted path and failing, progressively strip
the last component and retry. This handles `from lmdeploy.models import MyClass` where `MyClass`
is defined inside `__init__.py` — the import resolves to the `lmdeploy.models` package.

### 6.3 Stdlib and third-party filtering

There is no list of Python stdlib module names in the codebase, and maintaining one is fragile.
The implicit filter is the correct approach: if a dotted module path (after absolute-import
normalization or relative-import resolution) is not present in the project module index, it is not
an intra-project dependency, and no relation is emitted.

This correctly handles `import os`, `from typing import Optional`, `import torch`, and
`from transformers import AutoModel` — all absent from the project module index.

**Edge case**: a project module named `os` or `typing` (unlikely but valid). If such a file exists
in the project and is parsed, `generateModuleId()` will produce `os` or `typing` as its ID, and
it will appear in the index. An import of `import os` would then incorrectly resolve to the
project module. This is an unavoidable consequence of implicit filtering. A future improvement
could maintain a known-stdlib list, but this is out of scope for this change.

### 6.4 Star imports (`from module import *`)

`from lmdeploy.utils import *` is valid Python. The tree-sitter bridge currently captures the
`module_name` as `lmdeploy.utils`, with `items` left unset or as `[{ name: '*' }]` depending on
AST structure. The dependency relation target is the module, not the specific names imported, so
star imports are resolved identically to named imports. No special handling is needed.

### 6.5 Conditional imports and `TYPE_CHECKING` blocks

Python commonly uses:

```python
from __future__ import annotations
from typing import TYPE_CHECKING
if TYPE_CHECKING:
    from lmdeploy.models.base import BaseModel
```

Imports inside `if TYPE_CHECKING:` blocks are not top-level statements and are not captured by
the tree-sitter bridge (which only processes top-level children of the module node). This means
type-hint-only imports will be silently skipped. This is acceptable — these imports do not
represent runtime dependencies and their absence is correct for a package dependency graph focused
on runtime coupling.

### 6.6 Aliased imports (`import X as Y`) — bridge limitation requiring fix in resolution

The tree-sitter bridge reads the `name` field of `import_statement` verbatim. For
`import lmdeploy.models.base as base`, the `name` field points to the `aliased_import` node
whose text is `lmdeploy.models.base as base`. The bridge therefore stores `imp.module =
"lmdeploy.models.base as base"`.

The resolution logic in `createImportDependency()` must strip the ` as <identifier>` suffix
before performing the module index lookup:

```
rawModule = "lmdeploy.models.base as base"
  → stripped = rawModule.replace(/ as \w+$/, '')   → "lmdeploy.models.base"
  → look up "lmdeploy.models.base" in modulePathIndex
```

This normalisation must be applied to every `imp.module` value before any resolution step
(absolute lookup, relative resolution, or progressive-strip fallback). Failing to do so would
silently drop all aliased `import X as Y` edges.

**Note**: `from X import Y as Z` is unaffected — `module_name` contains only the module path
(`X`), not the alias.

### 6.7 Multi-name `import` statements — known bridge limitation (out of scope)

For `import os, sys`, tree-sitter returns multiple `dotted_name` children, but the bridge only
reads the first via `childForFieldName('name')`. The result is that only `os` is captured; `sys`
is silently dropped. This is a pre-existing bridge limitation. It affects only the bare
`import A, B` form (rare in typical Python projects, which favour `from X import Y`). Fixing it
would require changing `TreeSitterBridge.extractImport()` to iterate over all `dotted_name`
children and return multiple `PythonRawImport` records. This is out of scope for this change;
the limitation is noted here for completeness.

## 8. Test Strategy

### Unit tests (`tests/unit/plugins/python/archjson-mapper.test.ts`)

New test group: `mapModules — dependency relations`:

1. **Basic absolute import**: Two modules, one imports the other by dotted path. Verify one
   `dependency` relation with correct `source` (importer's module ID) and `target` (importee's
   module ID).

2. **Relative import (`.utils`)**: Source at `lmdeploy/models/base.py`, import `.utils`. Verify
   resolves to `lmdeploy.models.utils` if that module exists in the index.

3. **Relative import (`..common`)**: Source at `lmdeploy/models/base.py`, import `..common`.
   Verify resolves to `lmdeploy.common`.

4. **Stdlib import excluded**: Module with `import os` only. Verify zero dependency relations.

5. **Third-party import excluded**: Module with `from torch import Tensor`. Verify zero dependency
   relations.

6. **Import of `__init__.py` package**: Module imports `from lmdeploy.models import X` where X is
   defined in `lmdeploy/models/__init__.py`. Verify resolves to `lmdeploy.models`.

7. **Self-import guard**: Defensive test — module importing itself. Verify zero relations.

8. **Duplicate import deduplicated**: Same import appears twice in `PythonRawModule.imports`.
   Verify exactly one relation, not two.

9. **Three-module chain** (A → B → C): Verify two `dependency` relations, both correct.

10. **No imports**: Module with empty `imports` array. Verify zero new relations.

11. **No `workspaceRoot`**: Relative imports without workspace context. Verify relative imports
    are silently skipped; absolute imports that happen to match a module name by basename still
    resolve (or also skip gracefully if the path cannot be computed).

12. **Bare relative import (`from . import X`)**: `imp.module` is always `"."` (the empty-string
    case does not occur). Verify it resolves to the current package's `__init__.py` module ID if
    one exists, or is silently skipped if not.

13. **Both inheritance and dependency in same output**: A module with both a base-class import
    and an ordinary import. Verify both `inheritance` and `dependency` relations appear without
    collision.

14. **Aliased `import X as Y`**: Module with `imp.module = "lmdeploy.models.base as base"` (as
    produced by the bridge for aliased imports). Verify the ` as base` suffix is stripped and the
    relation resolves to `lmdeploy.models.base` correctly.

### Integration validation

**Manual smoke test**: Run `node dist/cli/index.js analyze -s <lmdeploy-root> --lang python -f
json` and verify the output `relations` array contains entries with `type: "dependency"` whose
`source` and `target` values match known module IDs (e.g. `lmdeploy.models.base` →
`lmdeploy.engine.async_llm_engine`). The package diagram must show `-->` arrows between nodes.

All existing `archjson-mapper.test.ts` and `python-plugin.test.ts` tests must remain green — the
new relations are strictly additive and replace the current stub (which produced unreachable
targets, not correct targets).

## Appendix: Data flow summary

```
TreeSitterBridge.parseCode("from lmdeploy.models.base import BaseModel", filePath)
  → PythonRawImport { module: "lmdeploy.models.base", items: [{name: "BaseModel"}] }
  → stored in PythonRawModule.imports[]

TreeSitterBridge.parseCode("from .utils import helper", filePath)
  → PythonRawImport { module: ".utils" }
  → stored in PythonRawModule.imports[]

mapModules(modules, workspaceRoot)
  PRE-PASS:
    → modulePathIndex = Map {
        "lmdeploy.models.base"   → "lmdeploy.models.base",
        "lmdeploy.models.utils"  → "lmdeploy.models.utils",
        "lmdeploy.engine"        → "lmdeploy.engine",   // __init__.py
        ...
      }

  PER-MODULE (filePath = "/project/lmdeploy/engine/async_engine.py"):
    sourceModuleId = generateModuleId(...) = "lmdeploy.engine.async_engine"

    import "lmdeploy.models.base":
      → absolute import, look up in index → FOUND "lmdeploy.models.base"
      → emit Relation { type: "dependency",
                        source: "lmdeploy.engine.async_engine",
                        target: "lmdeploy.models.base" }

    import ".utils":
      → relative import, dotCount=1, sourceDir="lmdeploy/engine", suffix="utils"
      → resolved dotted = "lmdeploy.engine.utils"
      → look up in index → FOUND "lmdeploy.engine.utils"
      → emit Relation { type: "dependency",
                        source: "lmdeploy.engine.async_engine",
                        target: "lmdeploy.engine.utils" }

    import "torch":
      → absolute import, look up in index → NOT FOUND (third-party)
      → skip, no relation emitted
```

**Note**: When `workspaceRoot` is undefined (single-file `parseCode()` path), `generateModuleId()`
falls back to the short module name, and relative import resolution is skipped. No dependency
relations are emitted in single-file mode, which is correct — there is no package graph context.
