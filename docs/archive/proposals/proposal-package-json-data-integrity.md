# Proposal: Package JSON Data Integrity

**Status**: Draft
**Date**: 2026-03-13
**Author**: ArchGuard Team

---

## 1. Background

### Current state

When ArchGuard generates JSON output (`-f json`) for a TypeScript project, the `overview/package.json` file always has `relations: []`, even though the Mermaid package diagram rendered from the same analysis shows hundreds of module-level dependency arrows. Additionally, class-level ArchJSON files have 453+ "undefined entity" warnings because relation source/target use bare class names while entity IDs use a qualified path format.

### Root cause

**Issue 1 — Package JSON missing relations:**

The package-level Mermaid diagram for TypeScript is generated from `archJSON.extensions.tsAnalysis.moduleGraph` (a `TsModuleGraph` with `nodes` and `edges` arrays). This happens in `src/cli/processors/diagram-output-router.ts`, `generateTsModuleGraphOutput()`, which calls `renderTsModuleGraph(moduleGraph)` to produce Mermaid code.

However, when `format === 'json'`, the router writes `archJSON` directly (line 83):

```typescript
if (format === 'json') {
  await fs.writeJson(paths.paths.json, canonicalizeArchJson(archJSON), { spaces: 2 });
  return;
}
```

The `archJSON.relations` array at this point contains class-level relations (composition, inheritance, dependency between classes). The `TsModuleGraph.edges` — the actual package-level module dependencies — are stored in `archJSON.extensions.tsAnalysis.moduleGraph.edges` and are never serialized into `archJSON.relations` for the package diagram.

**Issue 2 — Class relation ID format mismatch:**

The `RelationExtractor` in `src/parser/relation-extractor.ts` records relations using bare class names as `source` and `target`:

```typescript
// relation-extractor.ts line 206
relations.push({
  id: `${source}_${type}_${target}`,
  type,
  source,           // e.g. 'TypeScriptParser'
  target,           // e.g. 'ParallelParser'
});
```

Meanwhile, entity IDs use a qualified file-relative format generated by the TypeScript parser:

```
src/parser/typescript-parser.ts.TypeScriptParser
```

When the `ArchJSON.validator.validate()` checks `entityIds.has(relation.source)`, `TypeScriptParser` is not in the set — only `src/parser/typescript-parser.ts.TypeScriptParser` is. This produces 453 `DANGLING_RELATION_SOURCE` and `DANGLING_RELATION_TARGET` warnings, and means only ~3.5% of relations are usable by downstream tools.

### Impact

- MCP tools `get_dependencies` and `get_dependents` return empty results for any TypeScript package-level query because `relations: []` in `overview/package.json`.
- Class diagram queries via the MCP layer resolve 0 relations, making architecture analysis unreliable.
- The 453 undefined-entity warnings flood CI output on every `analyze -f json` run.

---

## 2. Proposed Solution

### Approach

**Fix 1 — Serialize TsModuleGraph edges into package ArchJSON relations:**

When `format === 'json'` and the ArchJSON has a `tsAnalysis.moduleGraph`, convert the `TsModuleGraph.edges` array into `Relation[]` and replace `archJSON.relations` before writing. Each edge `{ from, to, strength }` maps to:

```typescript
{
  id: `${from}_dependency_${to}`,
  type: 'dependency',
  source: from,   // module ID, e.g. 'src/parser'
  target: to,     // module ID, e.g. 'src/cli'
}
```

This is localized to `DiagramOutputRouter.route()` — a one-time transformation before the `writeJson` call when the TS module graph is present.

**Fix 2 — Qualify relation source/target IDs in RelationExtractor:**

After all entities are parsed, a post-pass maps bare class names to their qualified entity IDs. The entity list is available when `TypeScriptParser.parseProject()` assembles the final ArchJSON. Build a `Map<string, string>` from bare name to qualified ID, then rewrite `relation.source` and `relation.target` for each relation.

This is cleaner than modifying RelationExtractor itself (which operates per-file without global entity context).

### Key changes

| File | Change |
|---|---|
| `src/cli/processors/diagram-output-router.ts` | Before JSON write: when `tsAnalysis.moduleGraph` present at `level=package`, synthesize relations from edges |
| `src/parser/typescript-parser.ts` | Post-pass in `parseProject()`: build name→entityId index, rewrite relation source/target |
| `tests/unit/cli/processors/diagram-output-router.test.ts` | Tests for package JSON relation serialization |
| `tests/unit/parser/typescript-parser.test.ts` | Tests for relation ID qualification |

---

## 3. Acceptance Criteria

1. `overview/package.json` produced by `analyze -f json` has `relations` array with one entry per module-level dependency edge shown in the Mermaid diagram.
2. Each package relation has `source` and `target` matching node IDs present in the corresponding `entities` (or module node IDs from the TsModuleGraph).
3. Class-level ArchJSON has `relations` with `source` and `target` matching entity IDs (e.g., `src/parser/typescript-parser.ts.TypeScriptParser`).
4. The 453-warning flood drops to 0 `DANGLING_RELATION_SOURCE` / `DANGLING_RELATION_TARGET` for internal relations.
5. No regression in Mermaid diagram output (relations in `.mmd` files are unchanged).

---

## 4. Out of Scope

- Fixing relation IDs for Go, Java, Python, or C++ plugins (each has different ID conventions that may or may not have this problem).
- Adding package-level pseudo-entities to the class ArchJSON (would require type system changes).
- Serializing `TsModuleGraph.nodes` as entities in the package JSON (out of scope for this plan; current focus is relations).
